## Talk Structure (Current Short Version)

### Framing

- Topic is /cve-fix, but real value is applying skill design lessons broadly
- "Skill" = Claude skill: use Claude interactively to create automation that
  runs Claude non-interactively
- ~20 examples of agents in release management automation context
- Early lessons, seeking feedback from others

### Core Lessons

1. **New Domains of Automation** - Custom, complex automation that wasn't
   practical before is now possible to produce quickly and maintain. Still build
   on shared quality tools. Processes should be written-down-as-automation.
   Shared and custom skills both valuable.

2. **Design Around Indeterminism** - Extract as much work to deterministic
   logic as possible. Treat agent call like a function call for "fuzzy" logic
   that would be overly complex with traditional automation.

3. **Design the Context (more not always better)** - Interactive sessions
   benefit from more context. Non-interactive narrow-problem solvers suffer
   from unnecessary context causing unwanted inconsistencies.

### Design Patterns

1. **Pseudocode as Prototype** - Skills start as English in Markdown, run like
   code. Recommended starting point.
   - Phase 0: Mostly md
     - submariner/.agents/workflows/cve-fix.md
     - submariner-release-management/.agents/workflows/scan-cves.md
   - Phase 1: Mixed md/sh
     - skills/cve-fix/SKILL.md
   - Phase 2: Mostly sh
     - /add-release-notes -> scripts/release-notes/auto-apply.sh
     - /cve-fix -> WIP PR: shipyard/pull/2383
   - Phase 3: All sh, optional agent (wrapped by both make and agent)
     - scripts/create-component-release.sh +
       skills/create-component-release/SKILL.md +
       submariner-release-management/Makefile#L18
     - scripts/rpm-lockfile-update.sh +
       skills/rpm-lockfile-update/SKILL.md +
       submariner-release-management/Makefile#L23
     - (many more end in this state)

2. **Pulse-Agnostic Docs** - Create context for agents that's also docs for
   humans. Massively increases productivity of docs.
   - Example: /context skill
   - submariner-release-management/.agents/workflows
   - submariner-operator/.agents/workflows

3. **Small World, Many Agents** - Craft data to create focused context. Invoke
   many agents in parallel, each on a discrete problem.
   - /add-release-notes: collect.sh -> prepare.sh -> review.sh +
     review-prompt.md
   - /cve-fix: detect.sh -> fix-all.sh -> review.sh + review-prompt.md

4. **Proper Plans** - Effort into quality plans is massively productive. More
   human effort collaborating on plans before execution.
   - SEP-0031: modernize-enhancements -> enhancements/pull/267
   - SEP-0032: cve-fix-refactoring -> shipyard/pull/2383

1. Konflux builds 8 components across 5 upstream repos (operator,
   submariner, lighthouse, shipyard, subctl) via Tekton pipelines on
   release branches. Hermetic, multi-arch (x86_64, arm64, ppc64le, s390x),

6. FBC releases created for 6 OCP versions with exhaustive cross-validation
   (same bundle SHA, matching component SHAs across 10 sources per component).
7. `make get-fbc-urls` (463-line script) extracts catalog URLs for QE to
   install snapshot operator via standard OLM.

### Mapping to Design Patterns (for slides)

These are concrete examples from the release automation that illustrate
the design patterns already in the short talk:

- **New Domains of Automation**: the entire 20-step release lifecycle
  was manual before. 27 Make targets + 28 scripts + 15 skills now cover
  it. Eliminated midstream repo entirely.
- **Design Around Indeterminism**: bundle-image-update.sh is 536 lines
  of deterministic SHA extraction/validation — agent only invoked for
  the fuzzy parts like release notes review.
- **Design the Context**: release-status.sh (1,474 lines) as a
  dashboard that crafts context for both humans and agents to understand
  release state. 22 workflow docs serve both as agent context and human
  runbooks.
- **Pseudocode as Prototype**: 15 skills at various phases of the
  evolution. Many started as workflow docs and graduated to
  make-wrapped scripts.
- **Pulse-Agnostic Docs**: 22 agent workflow docs that are also the
  team's runbooks. CLAUDE.md with full 20-step release workflow.
- **Small World, Many Agents**: CVE fix scripts (detect -> fix-all ->
  review), release notes (collect -> prepare -> review), each with
  focused review-prompt.md for the agent step.
- **Proper Plans**: configure-downstream.sh generates 49 files across
  3 commits from a single command — extensive planning went into the
  Konflux onboarding design.

### Three-Entry-Point Architecture

Skills evolved to a pattern where every automation has three interfaces
to the same underlying script:

```text
Script (direct/CI) <-- scripts/bundle-image-update.sh
Make target (humans) <-- make bundle-image-update
Skill (Claude)       <-- /bundle-image-update
```

The skill is a thin wrapper that `exec`s the script. The Make target
adds parameter validation before calling the script. Both invoke the
same logic. This emerged from the April 15 refactoring (PRs #53-56).

### The April 15 Refactoring (PRs #53-56)

Four PRs merged in one day, all extracting inline SKILL.md logic to
standalone scripts:

- PR #53: configure-downstream
- PR #54: konflux-component-setup (1,484-line SKILL.md -> thin wrapper)
- PR #55: konflux-bundle-setup (889-line SKILL.md -> thin wrapper)
- PR #56: bundle-image-update (755-line SKILL.md -> thin wrapper)

All PRs were net-negative in line count (refactoring reduced code).
Established the three-entry-point pattern as the standard. Driven by
real pain: large inline skills were hard to test, debug, and couldn't
be used outside Claude.

### Release Notes Workflow Design (PR #49)

Best example of "Design Around Indeterminism" + "Small World, Many
Agents" combined:

1. Deterministic filtering: 27 Jira issues -> 6 candidates
2. Evidence pre-fetch: Jira details, DFBUGS, GitHub PRs, git logs
3. Per-issue agent spawning: one agent per issue, evaluates pre-fetched
   evidence (agent judges, does not search)
4. Default is KEEP; only removes cross-squad addon-only fixes
5. Each removal is a separate revertable commit
6. 20+ commits in PR refining the prompt — real prompt engineering

### Workflow Doc Structure

22 docs in .agents/workflows/ follow a consistent format:

- **Title** (H1)
- **When** — trigger condition (precondition for Claude)
- **Process** — what to do, which repo, exact commands
- **Done When** — verification criteria with bash commands

Referenced from CLAUDE.md using `@.agents/workflows/name.md` syntax.
The format works for both humans (runbook) and agents (instructions).

### Lessons Learned (from PRs and issues)

- **set -e danger** (PRs #52, #57): `set -e` causes scripts to exit
  at command substitutions before error-handling runs, silently
  swallowing diagnostics. Fix: use `&&`/`||` to capture exit codes.
  Hit this bug twice in verification scripts.
- **Inline skills unmaintainable at ~1,000 lines**: Led to the April 15
  refactoring wave. Skills that started as monolithic SKILL.md files
  couldn't be tested independently or used from CI.
- **Real releases expose real bugs**: Permission bugs (PR #37),
  snapshot filter issues (PR #30), API status changes (PR #57), version
  label skip causing all prod images tagged wrong (PR #60).

### Conductor Vision (Issue #64)

The capstone architecture: a meta-skill that orchestrates all others.

Key design decision: **stateless re-evaluation** instead of a state
machine. No loop tracking, no history, no state. The conductor
re-checks structured state each time and picks the next incomplete
step. Cycles (CVE scan, fix, rescan) are not tracked as loops — they
are just incomplete steps on re-evaluation.

Two components: (1) structured state checker outputting machine-readable
JSON, (2) conductor skill that reads JSON and selects next step.

Quotable: "The quality bar shifts entirely to the state checker's
thoroughness."

### Release Timing Data (PR #48)

- Y-stream release: 17-50 hours of work
- Z-stream release: 13-42 hours of work
- Calendar time: 1-3 weeks including QE approval
- Critical path: QE approval (3-14 days, external dependency)
- Highest risk step: CVE fixes (4-24 hours, highly variable)

### Plugin Distribution

The repo is structured as a Claude Code plugin with
`.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json`.
The entire skill set is installable via `/plugin marketplace add`,
making it distributable to other team members.

### Other Design Details

- **Dual-mode execution**: Every automated step works via both a
  Claude skill and a make target — AI-assisted or manual human workflow
- **Verification-first**: `make apply` runs `make test-remote` as a
  prerequisite — cannot apply a release without passing all validations
- **Agent decision points**: konflux-ci-fix has explicit
  "AGENT DECISION POINT" markers where Claude pauses for human input
- **Per-step commits**: konflux-component-setup creates per-step
  commits so reviewers can evaluate each change independently
- **Context: fork**: Some skills operate in fork repos, not the
  release management repo (marked with `context: fork` in frontmatter)
- **Security**: konflux-ci-fix verifies SHA-256 checksum of downloaded
  scripts before executing
