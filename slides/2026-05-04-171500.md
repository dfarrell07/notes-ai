## Talk Structure (Current Short Version)

### Framing

- Topic is /cve-fix, but real value is applying skill design lessons broadly
- "Skill" = Claude skill: use Claude interactively to create automation that
  runs Claude non-interactively
- ~20 examples of agents in release management automation context
- Early lessons, seeking feedback from others

### Core Lessons

1. **New Domains of Automation** - Custom, complex automation that wasn't
   practical before is now possible to produce quickly and maintain. Still build
   on shared quality tools. Processes should be written-down-as-automation.
   Shared and custom skills both valuable.

2. **Design Around Indeterminism** - Extract as much work to deterministic
   logic as possible. Treat agent call like a function call for "fuzzy" logic
   that would be overly complex with traditional automation.

3. **Design the Context (more not always better)** - Interactive sessions
   benefit from more context. Non-interactive narrow-problem solvers suffer
   from unnecessary context causing unwanted inconsistencies.

### Design Patterns

1. **Pseudocode as Prototype** - Skills start as English in Markdown, run like
   code. Recommended starting point.
   - Phase 0: Mostly md
     - submariner/.agents/workflows/cve-fix.md
     - submariner-release-management/.agents/workflows/scan-cves.md
   - Phase 1: Mixed md/sh
     - skills/cve-fix/SKILL.md
   - Phase 2: Mostly sh
     - /add-release-notes -> scripts/release-notes/auto-apply.sh
     - /cve-fix -> WIP PR: shipyard/pull/2383
   - Phase 3: All sh, optional agent (wrapped by both make and agent)
     - scripts/create-component-release.sh +
       skills/create-component-release/SKILL.md +
       submariner-release-management/Makefile#L18
     - scripts/rpm-lockfile-update.sh +
       skills/rpm-lockfile-update/SKILL.md +
       submariner-release-management/Makefile#L23
     - (many more end in this state)

2. **Pulse-Agnostic Docs** - Create context for agents that's also docs for
   humans. Massively increases productivity of docs.
   - Example: /context skill
   - submariner-release-management/.agents/workflows
   - submariner-operator/.agents/workflows

3. **Small World, Many Agents** - Craft data to create focused context. Invoke
   many agents in parallel, each on a discrete problem.
   - /add-release-notes: collect.sh -> prepare.sh -> review.sh +
     review-prompt.md
   - /cve-fix: detect.sh -> fix-all.sh -> review.sh + review-prompt.md

4. **Proper Plans** - Effort into quality plans is massively productive. More
   human effort collaborating on plans before execution.
   - SEP-0031: modernize-enhancements -> enhancements/pull/267
   - SEP-0032: cve-fix-refactoring -> shipyard/pull/2383

- Check results at end

### Three-Entry-Point Architecture

Skills evolved to a pattern where every automation has three interfaces
to the same underlying script:

```text
Script (direct/CI) <-- scripts/bundle-image-update.sh
Make target (humans) <-- make bundle-image-update
Skill (Claude)       <-- /bundle-image-update
```

The skill is a thin wrapper that `exec`s the script. The Make target
adds parameter validation before calling the script. Both invoke the
same logic. This emerged from the April 15 refactoring (PRs #53-56).

### The April 15 Refactoring (PRs #53-56)

Four PRs merged in one day, all extracting inline SKILL.md logic to
standalone scripts:

- PR #53: configure-downstream
- PR #54: konflux-component-setup (1,484-line SKILL.md -> thin wrapper)
- PR #55: konflux-bundle-setup (889-line SKILL.md -> thin wrapper)
- PR #56: bundle-image-update (755-line SKILL.md -> thin wrapper)

All PRs were net-negative in line count (refactoring reduced code).
Established the three-entry-point pattern as the standard. Driven by
real pain: large inline skills were hard to test, debug, and couldn't
be used outside Claude.

### Release Notes Workflow Design (PR #49)

Best example of "Design Around Indeterminism" + "Small World, Many
Agents" combined:

1. Deterministic filtering: 27 Jira issues -> 6 candidates
2. Evidence pre-fetch: Jira details, DFBUGS, GitHub PRs, git logs
3. Per-issue agent spawning: one agent per issue, evaluates pre-fetched
   evidence (agent judges, does not search)
4. Default is KEEP; only removes cross-squad addon-only fixes
5. Each removal is a separate revertable commit
6. 20+ commits in PR refining the prompt — real prompt engineering

### Workflow Doc Structure

22 docs in .agents/workflows/ follow a consistent format:

- **Title** (H1)
- **When** — trigger condition (precondition for Claude)
- **Process** — what to do, which repo, exact commands
- **Done When** — verification criteria with bash commands

Referenced from CLAUDE.md using `@.agents/workflows/name.md` syntax.
The format works for both humans (runbook) and agents (instructions).

### Lessons Learned (from PRs and issues)

- **set -e danger** (PRs #52, #57): `set -e` causes scripts to exit
  at command substitutions before error-handling runs, silently
  swallowing diagnostics. Fix: use `&&`/`||` to capture exit codes.
  Hit this bug twice in verification scripts.
- **Inline skills unmaintainable at ~1,000 lines**: Led to the April 15
  refactoring wave. Skills that started as monolithic SKILL.md files
  couldn't be tested independently or used from CI.
- **Real releases expose real bugs**: Permission bugs (PR #37),
  snapshot filter issues (PR #30), API status changes (PR #57), version
  label skip causing all prod images tagged wrong (PR #60).

### Conductor Vision (Issue #64)

The capstone architecture: a meta-skill that orchestrates all others.

Key design decision: **stateless re-evaluation** instead of a state
machine. No loop tracking, no history, no state. The conductor
re-checks structured state each time and picks the next incomplete
step. Cycles (CVE scan, fix, rescan) are not tracked as loops — they
are just incomplete steps on re-evaluation.

Two components: (1) structured state checker outputting machine-readable
JSON, (2) conductor skill that reads JSON and selects next step.

Quotable: "The quality bar shifts entirely to the state checker's
thoroughness."

### Open Question: Hosting and Secrets

The automation currently runs on a developer's laptop. This works but
doesn't scale. The challenge is the number of secrets required:

- Konflux/OpenShift cluster credentials (oc login)
- GitHub tokens (PR creation, repo access across 7+ repos)
- Jira credentials (release notes collection)
- Red Hat entitlements (/etc/pki/entitlement/*.pem for RPM lockfiles)
- Container registry auth (quay.io, registry.redhat.io)
- Signing keys (for release pipeline)
- Errata tool access

Hosting options to explore:

- CI/CD runner with secret injection (but interactive skills need
  human-in-the-loop)
- Dedicated release workstation with pre-configured credentials
- Vault-based secret management with on-demand injection
- Hybrid: deterministic scripts in CI, agent steps on laptop

### Open Question: Top-Level UI

Related to hosting — what should users interact with?

- Slack bot (trigger releases from chat, status updates in channel)
- MCP endpoint (expose skills as tool calls for any agent)
- CLI (current model — developer runs skills from terminal)
- Web dashboard (visual release status + skill triggers)
- Something else?

The conductor (Issue #64) needs a UI — the current model of running
Claude Code interactively on a laptop works for one person but not
for a team workflow.

### Key Motivator: Handoff to Sustainment

A major driver behind the automation and docs investment: make the
release process runnable by a sustainment team so core Submariner
contributors can move on to future projects. This explains why
several design patterns matter:

- **Pulse-Agnostic Docs**: workflow docs that serve both agents AND
  new human operators who didn't build the system
- **Three entry points**: sustainment team can use make targets
  (familiar) or skills (guided), same underlying logic
- **Written-Down-as-Automation**: the process IS the code, not
  tribal knowledge in someone's head
- **Plugin distribution**: the skill set is installable via marketplace,
  new team members get the full toolkit immediately

Both open questions (hosting/secrets, top-level UI) are also driven
by handoff — the current "run on my laptop" model is single-person.

### Release Timing Data (PR #48)

- Y-stream release: 17-50 hours of work
- Z-stream release: 13-42 hours of work
- Calendar time: 1-3 weeks including QE approval
- Critical path: QE approval (3-14 days, external dependency)
- Highest risk step: CVE fixes (4-24 hours, highly variable)

### Plugin Distribution

The repo is structured as a Claude Code plugin with
`.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json`.
The entire skill set is installable via `/plugin marketplace add`,
making it distributable to other team members.

### Other Design Details

- **Dual-mode execution**: Every automated step works via both a
  Claude skill and a make target — AI-assisted or manual human workflow
- **Verification-first**: `make apply` runs `make test-remote` as a
  prerequisite — cannot apply a release without passing all validations
- **Agent decision points**: konflux-ci-fix has explicit
  "AGENT DECISION POINT" markers where Claude pauses for human input
- **Per-step commits**: konflux-component-setup creates per-step
  commits so reviewers can evaluate each change independently
- **Context: fork**: Some skills operate in fork repos, not the
  release management repo (marked with `context: fork` in frontmatter)
- **Security**: konflux-ci-fix verifies SHA-256 checksum of downloaded
  scripts before executing

## CVE Fix Skill Architecture (from shipyard repo exploration)

The /cve-fix skill illustrates multiple patterns cleanly:

**Deterministic scripts:**

- detect.sh — parse repo/branch args, set up fix branch
- scan.sh — run grype, find CVEs
- fix-package.sh — `go get` update, exit 0=fixed, 2=needs review,
  3=persists
- fix-stdlib.sh — update Go version, exit 2 if minor version change
- locate.sh — find package in go.mod, check replace directives
- fix-all.sh — orchestrator: detect -> scan -> fix each CVE -> review

**Agent-driven scripts:**

- review.sh — invoke agent with pre-fetched evidence (scan results,
  fix summary, locate output, Go version), template-based prompt
- ignore.sh — agent decides to add CVE to .grype.yaml when fix
  requires breaking changes

**Key design details:**

- Exit codes communicate "I need agent help" (2=needs review)
- State file persists config across scripts
- Review prompt pre-fetches ALL evidence before invoking agent
- Multi-repo: spawns subagents via Bash (one per repo)
- Clean.sh handles orphaned processes/containers

## Web Research: Validation and New Ideas (2026-05-04)

### All Patterns Are Well-Validated

Every pattern in the talk has established industry support. None are
controversial or "dumb." The talk's contribution is applying them
together in a real production system and giving them actionable names.

### Established Names for the Patterns

| Talk Pattern | Established Name(s) |
| ------------ | ------------------- |
| Pseudocode as Prototype | Automation maturity model, graduated autonomy |
| Design Around Indeterminism | Guided determinism (Salesforce), hybrid reasoning |
| Design the Context | Context engineering (Anthropic) |
| Pulse-Agnostic Docs | Dual-audience documentation (no prior term for this) |
| Small World, Many Agents | Orchestrator-worker pattern, agents-as-tools |
| Proper Plans | Plan-then-execute (SAP, Goldman Sachs) |
| Three entry points (Phase 3) | Multiple entry points (not hexagonal — see note) |
| Stateless re-evaluation | Idempotent convergence (from IaC/Ansible/Terraform) |

### Strong Data Points to Reference

- **Context engineering** (Anthropic): non-interactive agents are
  single-shot (claude -p --print), so it's not about context degrading
  over time — it's minimal context by design, principle of least
  privilege applied to context
- **Plan-then-execute** (Goldman Sachs): 5-10x token reduction and
  3-4x faster execution vs ReAct for predictable workflows
- **Multi-agent** (Anthropic): orchestrator-worker system outperformed
  single-agent by 90.2%. Token usage explains 80% of variance.
- **Gartner**: 40% of agentic AI projects will be canceled by 2027
  due to costs, unclear value, or inadequate risk controls
- **Anthropic context engineering**: "finding the smallest possible
  set of high-signal tokens that maximize the likelihood of some
  desired outcome"

### Pushback to Prepare For

1. **"Agent becomes optional"** (Pseudocode as Prototype) — might
   sound like "agents are temporary." Counter: for well-understood
   stable tasks, pure code IS more reliable. The agent is for the
   fuzzy parts, and some tasks have no fuzzy parts.
2. **Reliability compounding** (Small World, Many Agents) — 95% per
   agent ^ 10 agents = 60% system reliability. Counter: each agent is
   narrowly focused with pre-fetched evidence, so per-agent reliability
   is higher than general-purpose agents.
3. **"What is fuzzy vs deterministic?"** (Design Around Indeterminism)
   — the boundary is itself an engineering judgment. It shifts as the
   domain becomes better understood.
4. **Planning rigidity** (Proper Plans) — community favors adaptive
   planning. Counter: the conductor pattern (stateless re-evaluation)
   provides adaptivity at the orchestration level.
5. **"Pulse-Agnostic" naming** — original term, audience may not parse
   immediately. Consider explaining "pulse" (human or silicon) upfront
   or using "dual-audience docs" as alternate framing.

### Patterns Worth Considering Adding

1. **Reflection/Self-Correction** — agent evaluates its own output
   and iterates. Peer to Proper Plans but post-execution.
2. **Harness Engineering** — the "harness" (code wrapping the agent)
   is where real complexity lives. All six patterns are aspects of
   harness design. Could be a framing for the whole talk.
3. **Effort Scaling** — explicitly encoding how much effort an agent
   should spend. Agents struggle to judge appropriate effort.

### For the Open Questions

**Hosting/Secrets:**

- Anthropic Managed Agents (public beta 2026): ephemeral containers +
  credential vaults + scoped permissions
- Industry consensus: "developer laptop" is prototyping, not production
- Short-lived credentials tied to agent identity and task context,
  not static injected secrets

**Top-Level UI:**

- Slack + MCP is the current front-runner (MCP tool calls grew 25x,
  Duolingo hit 250 weekly active users with Slack agent)
- Design principle: "embed into existing workflows, don't make users
  adopt new tools"
- CLI serves developers; Slack serves broader teams/sustainment

**Three Entry Points:**

- NOT Hexagonal Architecture — that's a stretch. Make target and
  skill are thin one-line passthroughs to the script, not independent
  adapters implementing abstract ports. Better framed as "multiple
  convenience entry points meeting users where they are" — CI calls
  script, humans type make, Claude uses skill. Same underlying logic.

**Stateless Re-Evaluation:**

- Maps to IaC idempotent convergence (Ansible/Terraform): scan current state,
  update to match desired state
- Anthropic's own guidance favors simpler patterns over complex
  frameworks: "find the simplest solution possible"

### Accuracy Notes (from codebase verification)

Verified against actual code — some references were overstated:

1. **"Parallel" in Small World** — the shell scripts themselves run
   sequentially, but the outer Claude agent spawns subagents in
   parallel (one per repo, one per issue). The parallelism is at
   the agent orchestration level, which is the intended meaning.
2. **SEPs as agent execution specs** — CONFIRMED. They are designed
   to be consumed by agents, like plan mode but extracted to a shared
   repo for human review. The plan IS the spec the agent follows.
3. **"Context rot"** — agents are single-shot (claude -p --print),
   no long context to rot. The real pattern is minimal context by
   design / principle of least privilege for context.
4. **Guided determinism** — CONFIRMED. fix-all.sh has zero mentions
   of "claude" or "agent." Fully deterministic pipeline with one
   review.sh call at the end. review-prompt.md is structured like
   a function signature with defined inputs and expected outputs.

### Broader Landscape (from deeper web research)

**Canonical frameworks to reference:**

- Andrew Ng's four agentic patterns (2024): Reflection, Tool Use,
  Planning, Multi-Agent Collaboration. Talk's patterns complement
  these — Ng doesn't cover deterministic-first, pseudocode evolution,
  context scoping, or dual-audience docs.
- Anthropic's "Building Effective Agents" (Dec 2024): "Find the
  simplest solution possible, and only increase complexity when
  needed." Five workflow patterns: prompt chaining, routing,
  parallelization, orchestrator-workers, evaluator-optimizer.
  Workflows (predefined paths) vs Agents (LLM-directed). The ethos
  of the whole talk.

**Strongest validation data points:**

- Vercel removed 80% of their agent's tools (15 down to 2).
  Accuracy: 80% -> 100%. Tokens: -37%. Speed: 3.5x faster.
  "Simple architecture isn't enough. The model needs good context.
  Invest in documentation, clear naming, and well-structured data."
- Vercel also found passive context (AGENTS.md) outperformed skills
  (on-demand context) in 56% of cases where skills weren't invoked.
- Datadog State of AI Engineering 2026: 69% of input tokens are
  system prompts (instructions, policy, tool guidance). Skill design
  IS the engineering.
- Agent Skills open standard: 32 competing tools adopted SKILL.md
  format within 48 hours of release. 100K GitHub stars.
- Anthropic 2026 report: projects with well-maintained context files
  see 40% fewer agent errors and 55% faster task completion.

**Talk's original contributions (not widely named elsewhere):**

- Pseudocode as Prototype — closest external match is "spec-driven
  development" (freeCodeCamp) and SudoLang. Not yet a named pattern.
  The four-phase graduation model is novel.
- Pulse-Agnostic Docs — "dual-audience documentation" exists as a
  concept but not as a formalized pattern. The When/Process/Done When
  structure is original.
- Both are design-time patterns. The talk's strength is design-time;
  the gap is runtime (see below).

**Additional patterns to be aware of (not in the talk, but relevant):**

- Evaluator-Optimizer / Reflection loop — agent reviews its own
  output and iterates. A core Ng/Anthropic pattern. The talk's
  review.sh is a single-pass version of this.
- Tool minimization / "Addition by subtraction" — Vercel's finding
  that fewer tools = better results. Counterintuitive but well-
  validated. Supports deterministic-first.
- Progressive autonomy — start with high human involvement, reduce
  as system proves itself. Related to but distinct from the
  human-in-the-loop lesson.
- Observability as first-class concern — 94% of production agent
  teams use full tracing (LangChain 2026). The talk doesn't cover
  runtime monitoring.
- Red/Green TDD with agents (Simon Willison) — write failing test,
  let agent make it pass. Constrains agent scope.

**Anti-patterns to warn against:**

- God Skill — monolithic skills that try to do everything (Block's
  principle: "decide what the agent should NOT do")
- Premature multi-agent — jumping to multi-agent before single agent
  reaches quality ceiling. "Bag of Agents" anti-pattern leads to
  17x error amplification.
- Firehose Effect — tools returning too much data, causing agent
  drift as conversations progress
- Negative instructions — "do not X" confuses agents; rephrase as
  "only X if"

**Design-time vs runtime gap:**

The talk's six patterns are strong on design-time (planning,
prototyping, context curation, documentation). Runtime concerns
(observability, resilience, cost control, eval gates) are a
separate topic — more relevant for systems you'll operate long-term
than ones you're packaging for handoff. Could be a future talk for
a different project.

## Comparison: openshift-eng/ai-helpers

Explored ~/ovnk/ai-helpers — a Claude Code plugin marketplace with
37 plugins maintained by the broader OpenShift engineering community.

### Design Patterns Worth Noting

- **Commands as AI instructions, not scripts**: Their rule is "if it
  could be a shell alias or Makefile target, it should not be a plugin
  command." This is the OPPOSITE of Phase 3. Interesting tension —
  ai-helpers commands are mostly fuzzy tasks (analyze disruption,
  groom tickets, suggest reviewers) so they stay as AI prompts.
  Submariner skills are mostly deterministic tasks with a fuzzy tail,
  so they graduate to scripts. Both are right for their context —
  validates "Design Around Indeterminism."
- **Semantic versioning enforced by CI**: PR validation checks that
  modified plugins bump version. Critical because Claude Code caches
  by version. Should adopt this.
- **claudelint**: A linting tool validating plugin structure,
  frontmatter, naming, marketplace registration. Should use for
  submariner-release-management.
- **Overlap detection**: utils:review-ai-helpers-overlap uses semantic
  matching to find duplicate commands before writing new ones.
- **Auto-generated docs**: CI regenerates PLUGINS.md and website data
  on any plugin change. Creates auto-PR.
- **Hooks for metrics/notifications**: Hook-only plugins with no
  commands — just event-driven behavior (session metrics, desktop
  notifications).
- **Reference docs pattern**: jira plugin has reference/*.md with MCP
  tool signatures, custom field mappings, JQL recipes. Good pattern
  for structured context.
