## Talk Structure (Current Short Version)

### Framing

- Topic is /cve-fix, but real value is applying skill design lessons broadly
- "Skill" = Claude skill: use Claude interactively to create automation that
  runs Claude non-interactively
- ~20 examples of agents in release management automation context
- Early lessons, seeking feedback from others

### Core Lessons

1. **New Domains of Automation** - Custom, complex automation that wasn't
   practical before is now possible to produce quickly and maintain. Still build
   on shared quality tools. Processes should be written-down-as-automation.
   Shared and custom skills both valuable.

2. **Design Around Indeterminism** - Extract as much work to deterministic
   logic as possible. Treat agent call like a function call for "fuzzy" logic
   that would be overly complex with traditional automation.

3. **Design the Context (more not always better)** - Interactive sessions
   benefit from more context. Non-interactive narrow-problem solvers suffer
   from unnecessary context causing unwanted inconsistencies.

### Design Patterns

1. **Pseudocode as Prototype** - Skills start as English in Markdown, run like
   code. Recommended starting point.
   - Phase 0: Mostly md
     - submariner/.agents/workflows/cve-fix.md
     - submariner-release-management/.agents/workflows/scan-cves.md
   - Phase 1: Mixed md/sh
     - skills/cve-fix/SKILL.md
   - Phase 2: Mostly sh
     - /add-release-notes -> scripts/release-notes/auto-apply.sh
     - /cve-fix -> WIP PR: shipyard/pull/2383
   - Phase 3: All sh, optional agent (wrapped by both make and agent)
     - scripts/create-component-release.sh +
       skills/create-component-release/SKILL.md +
       submariner-release-management/Makefile#L18
     - scripts/rpm-lockfile-update.sh +
       skills/rpm-lockfile-update/SKILL.md +
       submariner-release-management/Makefile#L23
     - (many more end in this state)

2. **Pulse-Agnostic Docs** - Create context for agents that's also docs for
   humans. Massively increases productivity of docs.
   - Example: /context skill
   - submariner-release-management/.agents/workflows
   - submariner-operator/.agents/workflows

3. **Small World, Many Agents** - Craft data to create focused context. Invoke
   many agents in parallel, each on a discrete problem.
   - /add-release-notes: collect.sh -> prepare.sh -> review.sh +
     review-prompt.md
   - /cve-fix: detect.sh -> fix-all.sh -> review.sh + review-prompt.md

4. **Proper Plans** - Effort into quality plans is massively productive. More
   human effort collaborating on plans before execution.
   - SEP-0031: modernize-enhancements -> enhancements/pull/267
   - SEP-0032: cve-fix-refactoring -> shipyard/pull/2383

## Data for Expanded Talk: Release Automation Overview (ACM-3314)

Source: ACM-3314 epic resolution comment. Comprehensive summary of the
Konflux migration and release management automation built for Submariner.

### The Big Picture: Snapshot Flow Architecture

1. Konflux builds 8 components across 5 upstream repos (operator,
   submariner, lighthouse, shipyard, subctl) via Tekton pipelines on
   release branches. Hermetic, multi-arch (x86_64, arm64, ppc64le, s390x),
   SBOM generation, Enterprise Contract compliance.
2. Each successful build produces a component snapshot with image SHAs.
3. `make bundle-image-update` (536-line script) queries Konflux for latest
   passing snapshot, extracts 7 component SHAs (8 with metrics-proxy),
   updates operator bundle manifests, verifies every SHA, commits.
4. Component Release YAMLs (9 components) generated, validated, applied to
   Konflux cluster. Release pipeline produces released images.
5. FBC catalog updated with new bundle SHA for all supported OCP versions
   (4.16-4.21, channels stable-0.17 through stable-0.24).
6. FBC releases created for 6 OCP versions with exhaustive cross-validation
   (same bundle SHA, matching component SHAs across 10 sources per component).
7. `make get-fbc-urls` (463-line script) extracts catalog URLs for QE to
   install snapshot operator via standard OLM.

### Scale of What Was Built

**submariner-release-management (stolostron/):**

- 27 Make targets automating full 20-step release lifecycle
- 28 shell scripts totaling 9,776 lines of automation
- 88 release YAML configs across versions 0.20-0.24 (stage + prod,
  component + FBC for 6 OCP versions)
- 5 GitHub Actions CI workflows (yaml, git, shell, markdown linting +
  release validation)
- 22 agent workflow docs as detailed runbooks for every release step
- 15 Claude Code skills as slash commands for the release workflow
- release-status.sh: 1,474-line dashboard checking all 20 steps
- CVE verification via Clair reports (make verify-cve-fixes)
- Release notes auto-collection from Jira with per-issue agent review
- configure-downstream.sh: one-shot Y-stream setup generating 49 files
  across 3 commits in konflux-release-data repo
- CVE scanning workflow spanning Go deps, RPM deps, Go toolchain across
  7 upstream repos

**submariner-operator-fbc (stolostron/):**

- File-Based Catalog supporting OCP 4.14-4.21, 8 channels (stable-0.17
  through stable-0.24)
- 12 Tekton pipelines (push + pull-request for OCP 4.16-4.21)
- update-bundle.sh (1,150+ lines) auto-discovers passing snapshots,
  validates test status, handles ADD/UPDATE/REPLACE scenarios
- Multi-arch builds, ImageDigestMirrorSet with automatic unreleased
  Y-stream rotation

**submariner-addon (stolostron/ .tekton/):**

- Tekton pipelines for multiple ACM release streams
- Hermetic, multi-arch builds with gomod prefetch
- OLM catalog artifacts with CRD/CSV content verification

**Shipyard CVE fix automation (submariner-io/shipyard):**

- /cve-fix skill works across any Submariner repo and branch
- 10 scripts (1,000 lines) in scripts/cve/: detect, scan, locate,
  fix-package, fix-stdlib, ignore, review (with AI agent review),
  fix-all orchestrator, test-lib
- Agent workflow doc (249-line 11-step runbook)
- Parallel execution across multiple repos and branches
- Integrated with grype scanning and Clair downstream verification

**Upstream repos with Konflux CI (5+ repos):**

- Tekton pipeline definitions, hermetic builds, multi-platform support
  committed directly to release branches
- RPM lockfile automation in submariner and shipyard repos

**Konflux platform integration:**

- konflux-release-data: tenant configs, ReleasePlan overlays,
  ReleasePlanAdmissions (deployed via ArgoCD)
- Enterprise Contract policy compliance
- Errata advisory pipeline integration
- Pyxis product metadata for Red Hat Catalog listings

### Mapping to Design Patterns (for slides)

These are concrete examples from the release automation that illustrate
the design patterns already in the short talk:

- **New Domains of Automation**: the entire 20-step release lifecycle
  was manual before. 27 Make targets + 28 scripts + 15 skills now cover
  it. Eliminated midstream repo entirely.
- **Design Around Indeterminism**: bundle-image-update.sh is 536 lines
  of deterministic SHA extraction/validation — agent only invoked for
  the fuzzy parts like release notes review.
- **Design the Context**: release-status.sh (1,474 lines) as a
  dashboard that crafts context for both humans and agents to understand
  release state. 22 workflow docs serve both as agent context and human
  runbooks.
- **Pseudocode as Prototype**: 15 skills at various phases of the
  evolution. Many started as workflow docs and graduated to
  make-wrapped scripts.
- **Pulse-Agnostic Docs**: 22 agent workflow docs that are also the
  team's runbooks. CLAUDE.md with full 20-step release workflow.
- **Small World, Many Agents**: CVE fix scripts (detect -> fix-all ->
  review), release notes (collect -> prepare -> review), each with
  focused review-prompt.md for the agent step.
- **Proper Plans**: configure-downstream.sh generates 49 files across
  3 commits from a single command — extensive planning went into the
  Konflux onboarding design.
