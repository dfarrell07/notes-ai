# Non-Go Linting and Validation Tools for Kubernetes Operator Repos

Comprehensive research on the best modern linting and validation tools for every
non-Go content type in a Kubernetes operator repository. Covers 12 content type
categories with tool comparisons, version info, CLI usage, and CI recommendations.

## 1. YAML Files

### yamllint -- Generic YAML Syntax/Style

- **Version**: 1.38.0
- **What it catches**: Syntax errors, key repetition, line length, trailing spaces,
  indentation, cosmetic issues
- **CLI**: `yamllint -c .yamllint.yml .`
- **GHA**: Yes -- `pip install yamllint && yamllint .`
- **Worth adding**: YES -- baseline requirement for any repo with YAML files
- **Limitation**: Does NOT validate Kubernetes schema or semantics, only generic
  YAML structure
- **Source**: [github.com/adrienverge/yamllint](https://github.com/adrienverge/yamllint)

### kubeconform -- K8s Manifest Schema Validation (WINNER over kubeval)

- **Version**: Actively maintained (kubeval is abandoned since 2020, only supports
  K8s 1.18.1 and below)
- **What it catches**: Invalid resource types, wrong field types, missing required
  fields against K8s OpenAPI specs
- **CLI**: `kubeconform -summary -strict -kubernetes-version 1.30.0 manifests/`
- **GHA**: Yes -- native Go binary, runs in about 7s on 50K+ resources (vs kubeval
  at 35s+)
- **CRD support**: Yes, via configurable schema locations or Datree CRDs-catalog
- **Output formats**: TAP, JUnit, JSON
- **Worth adding**: YES -- kubeval is dead, kubeconform is the clear successor
- **Source**: [github.com/yannh/kubeconform](https://github.com/yannh/kubeconform)

### kube-linter -- Security-Focused K8s Manifest Linting

- **Maintainer**: StackRox (Red Hat)
- **What it catches**: Running as root, missing resource limits, missing probes,
  `latest` tag, missing read-only rootfs, sensitive data not in secrets -- 40+
  built-in checks
- **CLI**: `kube-linter lint manifests/` or pipe kustomize output:
  `kustomize build overlays/prod | kube-linter lint -`
- **GHA**: Yes -- official GHA available
- **Configurable**: Enable/disable checks, create custom checks
- **Worth adding**: YES -- catches security issues kubeconform misses (kubeconform
  only checks schema, kube-linter checks best practices)
- **Source**: [github.com/stackrox/kube-linter](https://github.com/stackrox/kube-linter)

### Kustomize Overlay Validation

- **Primary approach**: `kustomize build overlays/<env>` in CI -- if it fails, the
  overlay is broken
- **Schema validation of output**: Pipe into kubeconform or kube-linter:
  `kustomize build overlays/prod | kubeconform -summary`
- **Specialized tools**:
  - kustomize-lint (0daryo) -- catches wrong image tags in production overlays
  - kustomize-wrapper (Python) -- wraps kustomize build + kubeval
- **Worth adding**: YES -- `kustomize build | kubeconform` is the recommended
  pattern

## 2. Dockerfiles

### hadolint -- Still the Best

- **Version**: v2.14.0 (September 2025)
- **What it catches**: Dockerfile best practices + ShellCheck integration for RUN
  instructions, LABEL validation, `latest` tag, missing USER instruction, apt-get
  caching
- **CLI**: `hadolint Dockerfile` or
  `docker run --rm -i hadolint/hadolint < Dockerfile`
- **GHA**: Yes -- official `hadolint/hadolint-action`
- **Rules**: DL-prefixed (hadolint native) + SC-prefixed (ShellCheck for inline
  bash)
- **Worth adding**: YES -- unchallenged leader for Dockerfile linting
- **Source**: [github.com/hadolint/hadolint](https://github.com/hadolint/hadolint)

### Complementary Tools

- **Dockle** -- lints built container IMAGES (not Dockerfiles), security-focused
- **Trivy** -- broader scope: vulnerabilities + misconfigurations in images, IaC,
  repos

## 3. Shell Scripts

### ShellCheck -- Unchallenged Standard

- **Version**: v0.11.0 (August 2025)
- **What it catches**: Syntax issues, semantic problems, quoting bugs, portability
  issues, deprecated constructs
- **CLI**: `shellcheck scripts/*.sh`
- **GHA**: Yes -- multiple GHA wrappers, also integrated into actionlint
- **Output formats**: JSON, CheckStyle XML, GCC-compatible, human readable
- **Worth adding**: YES -- no real alternative for static analysis of shell scripts
- **Source**: [github.com/koalaman/shellcheck](https://github.com/koalaman/shellcheck)

### shfmt -- Shell Formatter (Complement to ShellCheck)

- **What it does**: Auto-formats shell scripts (like `go fmt` for Go)
- **CLI**: `shfmt -d scripts/` (diff mode) or `shfmt -w scripts/` (write mode)
- **GHA**: Yes -- available via pre-commit
- **Worth adding**: YES as a complement to ShellCheck -- use both together

### bats-core -- Shell Script Testing

- **Version**: v1.13.0 (November 2024)
- **What it does**: TAP-compliant testing framework for Bash 3.2+
- **CLI**: `bats test/` -- runs all `.bats` files
- **Key features**: Parallel execution, tagging system (since v1.8.0), fail-fast
  (`--abort`), negative filtering
- **GHA**: Yes -- runs natively
- **Worth adding**: YES if you have non-trivial shell scripts worth testing
- **Source**: [github.com/bats-core/bats-core](https://github.com/bats-core/bats-core)

### Shellharden -- Supplement

- **What it does**: Detects quoting flaws and can AUTO-FIX them (unlike ShellCheck
  which only reports)
- **Worth adding**: OPTIONAL -- ShellCheck is sufficient for most projects

## 4. Markdown

### markdownlint-cli2 -- Recommended over markdownlint-cli

- **Version**: 0.22.1 (latest)
- **Author**: Same as the markdownlint library (David Anson) -- markdownlint-cli
  is by a different author
- **What it catches**: Heading levels, list indentation, link formatting,
  whitespace, trailing spaces, line length
- **CLI**: `markdownlint-cli2 '**/*.md'`
- **Config**: `.markdownlint-cli2.yaml` -- also used by vscode-markdownlint
  (shared config)
- **Key advantages over markdownlint-cli**: Configuration-first design, async
  performance, same author as the library
- **GHA**: Yes
- **Worth adding**: YES -- markdownlint-cli2 is the modern standard
- **Source**: [github.com/DavidAnson/markdownlint-cli2](https://github.com/DavidAnson/markdownlint-cli2)

### Link Checking -- lychee (Recommended)

- **Lychee** -- recommended replacement for deprecated markdown-link-check
  (gaurav-nelson)
  - Written in Rust, very fast
  - Checks Markdown, HTML, text files
  - GHA available: `lycheeverse/lychee-action`
  - Supports SARIF output, configurable exclusions
- **Alternatives**: Linkspector (official recommended replacement), mlc (Rust),
  linkinator (Node.js)
- **Worth adding**: YES -- broken links in docs degrade quality silently
- **Source**: [github.com/lycheeverse/lychee-action](https://github.com/lycheeverse/lychee-action)

## 5. GitHub Actions Workflow Files

### actionlint -- Primary GHA Linter

- **Version**: v1.7.12 (latest, 2026)
- **What it catches**: Syntax validation, strong type checking of `${{ }}`
  expressions, action input/output validation, reusable workflow checking, shell
  injection detection, deprecated command usage, runner label validation, cron
  syntax, glob syntax
- **CLI**: `actionlint` (auto-detects `.github/workflows/`)
- **Integrations**: Pipes `run:` scripts through ShellCheck automatically, Python
  scripts through pyflakes
- **GHA**: Yes -- official action, VS Code extension, pre-commit hook
- **Worth adding**: YES -- catches real bugs and security issues
- **Source**: [github.com/rhysd/actionlint](https://github.com/rhysd/actionlint)

### zizmor -- GHA Security Scanner (NEW, Highly Recommended)

- **What it catches**: Template injection vulnerabilities, unpinned action
  versions, mutable tags, secrets exposure, credential leaks, supply chain risks
  -- 24 audit rules
- **CLI**: `zizmor .github/workflows/`
- **Output**: SARIF (integrates with GitHub Advanced Security tab)
- **GHA**: Yes -- `zizmorcore/zizmor-action`
- **Real-world adoption**: Grafana Labs deployed across 2000+ repos after a
  security incident; academic paper (March 2026) found it has the widest coverage
  of any general-purpose GHA scanner
- **Worth adding**: YES -- fills security gaps actionlint does not cover (unpinned
  SHA hashes, secrets in env vars)
- **Source**: [github.com/zizmorcore/zizmor](https://github.com/zizmorcore/zizmor)

### Recommended: Use BOTH actionlint + zizmor

- actionlint = structural correctness + type safety
- zizmor = security posture + supply chain integrity

## 6. Makefiles

### checkmake -- Best Available

- **Version**: Actively maintained (updated March 2026), about 1.2k GitHub stars
- **What it catches**: Missing `.PHONY`, missing `all` target, missing help
  targets, configurable rules
- **CLI**: `checkmake Makefile`
- **Config**: `checkmake.ini` for rule configuration
- **GHA**: Available via pre-commit, also embedded in MegaLinter
- **Worth adding**: OPTIONAL -- useful but low priority compared to other linters
- **Source**: [github.com/checkmake/checkmake](https://github.com/checkmake/checkmake)

### Alternatives

- **unmake** (Rust) -- POSIX compliance checking
- **mbake** (Python) -- formatter + linter + VSCode extension, but mixed reviews
  and immature

## 7. Tekton Pipeline YAML

### IBM/tekton-lint -- Only Dedicated Tekton Linter

- **What it catches**: Invalid parameter names/types, missing task references,
  external task validation, undefined resource references
- **CLI**: `npx @ibm/tekton-lint@latest 'pipelines/**/*.yaml'`
- **Config**: `.tektonlintrc.yaml` for rules and external task definitions
- **Watch mode**: Available for development
- **Limitation**: Requires Node.js (npm/npx)
- **Worth adding**: YES if you have Tekton pipelines -- it catches errors that
  generic YAML linters miss
- **Source**: [github.com/IBM/tekton-lint](https://github.com/IBM/tekton-lint)

### Complementary: kubeconform + kube-linter

- Tekton resources are still K8s resources, so kubeconform and kube-linter can
  validate the K8s structure
- However they will not understand Tekton-specific semantics (parameter
  references, task dependencies, etc.)

## 8. OLM Bundle Manifests

### operator-sdk bundle validate -- The Standard

- **CLI**: `operator-sdk bundle validate ./bundle`
- **Default validators**: Bundle spec compliance (CSV, CRDs, metadata structure)
- **Optional validator suites**:
  - `--select-optional suite=operatorframework` -- full Operator Framework
    validation
  - `--select-optional name=operatorhub` -- OperatorHub.io metadata validation
  - `--select-optional name=good-practices` -- general best practices
  - Community validator -- community catalog criteria
  - AlphaDeprecatedAPIsValidator -- K8s API deprecation detection
- **K8s version targeting**: `--optional-values=k8s-version=1.30`
- **Multi-platform validation**: Checks CSV platform labels match manager image
  architectures
- **GHA**: Yes -- can be run in any CI
- **Worth adding**: YES -- mandatory for any operator that ships to OperatorHub or
  Red Hat catalogs

No significant alternatives exist -- operator-sdk bundle validate is the standard
for OLM bundles.

## 9. License Headers

### Apache SkyWalking Eyes (license-eye) -- Most Comprehensive

- **Version**: 0.7.0 (latest)
- **What it does**: Check AND fix license headers + resolve dependency licenses
- **CLI**: `license-eye header check` / `license-eye header fix` /
  `license-eye dependency resolve`
- **Config**: `.licenserc.yaml`
- **Multi-language**: Yes -- any file type
- **Dependency license auditing**: Go, Maven, npm, pip, Gemfile
- **GHA**: Yes -- native `apache/skywalking-eyes/header` and
  `apache/skywalking-eyes/dependency`
- **PR commenting**: Built-in (on-failure, always, or never)
- **Worth adding**: YES for multi-language repos -- the most feature-rich option
- **Source**: [github.com/apache/skywalking-eyes](https://github.com/apache/skywalking-eyes)

### google/addlicense -- Simplest

- **CLI**: `addlicense -check -c 'Red Hat' -l apache .` (check) or
  `addlicense -c 'Red Hat' -l apache .` (fix)
- **SPDX support**: Yes via `-s` flag
- **Multi-language**: Yes -- scans recursively by file extension
- **Pre-commit**: Yes
- **Limitation**: Cannot update/remove headers, limited comment style
  customization
- **Worth adding**: YES for simple use cases
- **Source**: [github.com/google/addlicense](https://github.com/google/addlicense)

### goheader (via golangci-lint) -- Go-Only

- **What it does**: Check license headers in .go files only
- **Config**: `.golangci.yml` -- supports `const` and `regexp` templates
- **Limitation**: Go files only, no fix mode
- **Worth adding**: ONLY if you already use golangci-lint and want header checking
  integrated there

### Recommendation for License Headers

- **Multi-language repo**: Use SkyWalking Eyes -- covers all file types +
  dependency license auditing
- **Go-only repo**: goheader via golangci-lint is sufficient for .go files
- **Simple needs**: addlicense is the lightest-weight option

## 10. Git Commits

### conventionalcommit/commitlint (Go) -- BEST for Go Projects

- **CLI**: `commitlint lint` or as a git hook
- **Config**: `.commitlint.yaml` (created via `commitlint config create`)
- **What it catches**: header length, body line length, type enum
  (feat/fix/docs/etc.), conventional commits format
- **No external runtime**: Pure Go binary, no Node.js or Python needed
- **Custom rules**: Implement Go `Rule` interface
- **Worth adding**: YES -- fits naturally in Go toolchains
- **Source**: [github.com/conventionalcommit/commitlint](https://github.com/conventionalcommit/commitlint)

### commitlint (JavaScript, @commitlint/cli) -- Most Popular Overall

- Requires Node.js -- adds unwanted dependency to a Go project
- Largest ecosystem, but overkill for Go projects

### gitlint (Python) -- Language-Neutral

- Requires Python -- adds unwanted dependency
- General-purpose git message linter, not specifically built for conventional
  commits

### Recommendation for Go operator repos

Use the Go-native `conventionalcommit/commitlint`.

## 11. RBAC Manifests

### rbac-police (Palo Alto Networks / Unit 42)

- **What it does**: Analyzes RBAC posture, identifies overprivileged service
  accounts, privilege escalation paths
- **Limitation**: Primarily designed for running clusters, not static manifest
  analysis

### Checkov (Bridgecrew)

- **What it does**: IaC scanner with Kubernetes-specific checks including RBAC
- **CLI**: `checkov -d manifests/` -- includes checks for excessive RBAC
  permissions
- **Worth adding**: OPTIONAL -- if you want static RBAC validation in CI

### kube-linter (already recommended above)

- Includes some RBAC-related checks as part of its security scanning

### Recommendation for RBAC Validation

- For static CI linting of RBAC manifests: kube-linter (already in the stack) +
  Checkov for deeper policy checks
- For cluster-level RBAC auditing: rbac-police
- There is NO dedicated standalone RBAC manifest linter -- the best approach is
  layering kube-linter + Checkov + manual review

## 12. Go Module Files (go.mod, go.sum)

### govulncheck -- RECOMMENDED Primary Tool

- **Official**: Maintained by the Go security team
- **Key advantage**: Symbol-level reachability analysis -- only flags
  vulnerabilities in code paths actually invoked
- **CLI**: `govulncheck ./...`
- **Database**: Go Vulnerability Database (curated, multi-source: NVD, GitHub
  Advisories, maintainer reports)
- **IDE integration**: Built into gopls/VS Code
- **Binary scanning**: Supported
- **Limitation**: No way to suppress/ignore findings
- **Worth adding**: YES -- should be the primary vulnerability scanner for any Go
  project

### nancy (Sonatype) -- Complementary

- **CLI**: `go list -json -deps ./... | nancy sleuth`
- **Database**: Sonatype OSS Index (different from Go vuln DB)
- **Key advantage**: Supports `.nancy-ignore` for suppressing known false
  positives
- **Limitation**: Module-level scanning only (more false positives), does not
  support non-semver deps
- **Worth adding**: OPTIONAL -- use alongside govulncheck for broader coverage
  from different data source

### govulncheck vs nancy summary

- govulncheck = fewer false positives (reachability analysis), official Go team
  backing
- nancy = different vulnerability database, suppression support
- Use govulncheck as primary, nancy as optional supplement

### go-mod-outdated -- Dependency Staleness

- **CLI**: `go list -u -m -json all | go-mod-outdated -direct -ci`
- **Last release**: February 2023 -- still functional but not actively developed
- **What it catches**: Shows outdated direct/indirect dependencies with available
  updates
- **CI mode**: `-ci` flag exits non-zero if outdated deps found
- **Worth adding**: OPTIONAL -- useful for periodic staleness checks, not blocking
  in CI
- **Source**: [github.com/psampaz/go-mod-outdated](https://github.com/psampaz/go-mod-outdated)

### go-licenses -- License Scanning of Go Dependencies

- **Version**: v2.0.0
- **CLI**: `go-licenses check ./...` (check forbidden licenses) /
  `go-licenses report ./...` /
  `go-licenses save ./... --save_path=third_party`
- **What it does**: Classifies licenses of all Go dependencies, can save license
  texts for redistribution compliance
- **Worth adding**: YES for any project shipping binaries -- needed for license
  compliance
- **Source**: [github.com/google/go-licenses](https://github.com/google/go-licenses)

## Recommended Tool Stack -- Priority Order

### Tier 1 -- Must Have (add to CI immediately)

1. **yamllint** -- baseline YAML syntax
2. **kubeconform** -- K8s manifest schema validation
3. **kube-linter** -- K8s security best practices
4. **hadolint** -- Dockerfile linting
5. **ShellCheck** -- shell script analysis
6. **actionlint** -- GHA workflow validation
7. **govulncheck** -- Go vulnerability scanning
8. **operator-sdk bundle validate** -- OLM bundle compliance

### Tier 2 -- Strongly Recommended

1. **zizmor** -- GHA security scanning
2. **markdownlint-cli2** -- Markdown linting
3. **shfmt** -- shell script formatting
4. **SkyWalking Eyes** or **addlicense** -- license header compliance
5. **go-licenses** -- dependency license scanning
6. **conventionalcommit/commitlint** (Go) -- commit message enforcement

### Tier 3 -- Nice to Have

1. **lychee** -- Markdown broken link checking
2. **checkmake** -- Makefile linting
3. **IBM/tekton-lint** -- Tekton pipeline validation (if using Tekton)
4. **nancy** -- supplemental vulnerability scanning (different DB than
    govulncheck)
5. **bats-core** -- shell script testing (if you have complex scripts)
6. **go-mod-outdated** -- dependency staleness monitoring
7. **Checkov** -- deeper RBAC/IaC policy checking
