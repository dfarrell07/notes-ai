# OVN-Kubernetes K8s Rebase Automation

## Summary

When Kubernetes releases a new version, OVN-Kubernetes must bump all
`k8s.io/*` dependencies across its three Go modules. This "K8s rebase"
happens roughly 3 times a year and currently takes 2-3 weeks of human
effort. Analysis of recent rebases (1.33 through 1.35) shows the work
is approximately 70% mechanical: dependency updates, vendoring, code
generation, and version string replacements across CI/docs. The
remaining 30% is fixing build and test breakage caused by upstream API
changes.

This document describes automation that handles the mechanical portion
and guides an agent through the breakage fixes using a pattern database
built from past rebases.

## Prior Art

The 1.35 rebase (upstream PR #6099, Meina-rh, merged 2026-03-19)
touched 2000+ files with 160k+ lines changed, but only ~150 lines
were human-authored code fixes. The rest was vendored dependencies,
generated code, and go.mod/go.sum churn.

The 1.33 rebase was done by Arti Sood (CORENET-5959, July 2025). The
1.34 by Meina-rh (October 2025). Patryk Diak (kyrtapz) contributed
fixup commits for 1.35.

## What the Repo Looks Like

Three independent Go modules, no `go.work` file:

| Module | Role | Vendors? | Current k8s |
| -------------------- | ---------------------- | -------- | ----------- |
| `go-controller/` | Main ovnk controller | Yes | v0.35.1 |
| `test/e2e/` | End-to-end tests | No | v0.35.1 |
| `test/conformance/` | NetworkPolicy conformance | No | v0.35.1 |

The `go-controller/` vendor directory contains ~3,200 `k8s.io/*.go`
files. This is why the go-controller commit alone changes 2000+ files.

The module path recently changed from `github.com/ovn-org/ovn-kubernetes`
to `github.com/ovn-kubernetes/ovn-kubernetes` as part of CNCF incubation
(PR #5935).

## How a Rebase Actually Plays Out

The commit sequence from the 1.35 rebase (PR #6099) reveals the real
workflow. It is not a clean pipeline — fixups are interleaved with
module updates:

```text
5238d48f7  2026-02-26  disable WatchListClient feature gate for unit tests
4b266dedb  2026-02-26  fix CNI utils test timeout after K8s 1.35 bump
d10c28ee8  2026-03-11  test/e2e rebase
2f6c248f2  2026-03-11  test/conformance rebase
32077814e  2026-03-11  Updates kubernetes versions
88885c3a0  2026-03-11  Fix e2e tests failure
4621b77f4  2026-03-12  go-controller rebase
baa2c262d  2026-03-12  update go mod tidy in e2e
9255baafc  2026-03-13  Bump codegen
```

Key observations:

- **Preemptive fixups came first.** The WatchListClient and timeout
  fixes were committed before any module was rebased. The author knew
  these issues would arise and fixed them upfront.
- **go-controller was done last**, not first. The test modules were
  rebased first (lighter-weight, faster feedback).
- **The go-controller commit included lint fixes inline** in its commit
  message body — 6 deprecation fixes described alongside the `go get`
  recipe.
- **A cleanup pass followed** (`update go mod tidy in e2e`) to catch
  residual dependency drift.

The 1.34 rebase used a slightly different commit message convention:
`"go-controller rebase 1.34"` with a `> go-controller` body prefix,
while 1.35 used `"go-controller rebase"` with no version in the title.
The automation should match whichever convention the team settles on.

## What to Automate

### Mechanical Steps (deterministic, scriptable)

**Module dependency updates** — for each module, run a series of
`go get` commands to pin k8s.io packages to the new version, then
`go mod tidy` (and `go mod vendor` for go-controller). The exact
commands per module, taken from the 1.35 rebase commits:

**go-controller** (16 `go get` commands):

```text
go get k8s.io/api@v0.35.1
go get k8s.io/client-go@v0.35.1
go get k8s.io/component-helpers@v0.35.1
go get k8s.io/kubernetes@v1.35.1
go get k8s.io/apiextensions-apiserver@v0.35.1
go get sigs.k8s.io/controller-runtime@v0.23.1
go get k8s.io/controller-manager@v0.35.1
go get sigs.k8s.io/structured-merge-diff/v4@v4.7.0
go get sigs.k8s.io/knftables@v0.0.20
go get k8s.io/kubelet@v0.35.1
go get sigs.k8s.io/apiserver-network-proxy/konnectivity-client@v0.34.0
go get k8s.io/kube-openapi
go get sigs.k8s.io/json
go get k8s.io/utils
go get github.com/openshift/api
go get github.com/openshift/client-go
go mod tidy
go mod vendor
```

Note: `apiextensions-apiserver` and `controller-manager` are `// indirect`
in go.mod but must be explicitly pinned because `go mod tidy` alone does
not reliably resolve them to the correct version.

Note: `structured-merge-diff` was referenced as `v4` in the `go get`
command but `go mod tidy` resolved it to `v6` — the major version
changed during the 1.34 rebase. The script should detect the correct
major version from the upstream k8s go.mod.

**test/e2e** (14 `go get` commands):

```text
go get k8s.io/kubernetes@v1.35.1
go get k8s.io/api@v0.35.1
go get k8s.io/client-go@v0.35.1
go get k8s.io/pod-security-admission@v0.35.1
go get k8s.io/kubectl@v0.35.1
go get sigs.k8s.io/controller-runtime@v0.23.1
go get k8s.io/mount-utils@v0.35.1
go get k8s.io/cri-client@v0.35.1
go get k8s.io/dynamic-resource-allocation@v0.35.1
go get k8s.io/kube-scheduler@v0.35.1
go get k8s.io/csi-translation-lib@v0.35.1
go get k8s.io/cloud-provider@v0.35.1
go get k8s.io/controller-manager@v0.35.1
go get k8s.io/kube-openapi
go mod tidy
```

Note: Many of these (mount-utils, cri-client, etc.) are transitive
dependencies that need explicit version pinning even though they do not
appear in the primary `require` block. Without explicit `go get`, `go
mod tidy` may resolve them to incompatible versions.

**test/conformance** (4 `go get` commands):

```text
go get k8s.io/kubernetes@v1.35.1
go get k8s.io/client-go@v0.35.1
go get k8s.io/kube-openapi
go get k8s.io/utils
go mod tidy
```

Note: This module uses `controller-runtime v0.22.1` while the other
two use `v0.23.1`. The rebase recipe did not explicitly bump it — `go
mod tidy` handles the resolution. The automation should not force
alignment.

**Code generation** — update the version pin in
`go-controller/hack/update-codegen.sh` (the line containing
`k8s.io/code-generator/cmd/%s@v0.XX.Y`), then run `make -C
go-controller codegen`. This regenerates ~90 files (CRD clients,
informers, listers, apply configurations).

**Version reference updates** — sed-based replacements using the old
version as the search pattern (not line numbers, which shift):

K8S_VERSION (6 files):

- `.github/workflows/test.yml`
- `.github/workflows/performance-test.yml`
- `contrib/kind-common.sh`
- `test/scripts/install-kind.sh`
- `test/scripts/upgrade-ovn.sh`
- `docs/developer-guide/local_testing_guide.md`

Go version (only if changed, 6 files):

- `go-controller/Makefile`
- `dist/images/Makefile`
- `.github/workflows/docker.yml`
- 3 `go.mod` files (`go` + `toolchain` directives)

Docs (2 files):

- `docs/features/requirements.md`
- `docs/installation/launching-ovn-kubernetes-on-kind.md`

### Agent-Guided Steps (require judgment)

After the mechanical steps, build and lint the go-controller module.
Parse errors against a pattern database of known breakage categories.
Apply mechanical fixes where possible, present the rest to the user.

## Breakage Pattern Database

10 patterns from analyzing the 1.31-1.35 rebases:

| ID | Pattern | 1.35 Example |
| --------------- | -------------------------------- | ---------------------------------------- |
| FG-DEFAULT | Feature gate default changed | WatchListClient enabled, hangs fake informers |
| FUNC-RENAME | Function renamed | WaitForServiceEndpointsNum to WaitForEndpointCount |
| SIG-CHANGE | Function signature changed | IsNodeReady now requires a logger parameter |
| ADMISSION-API | Admission webhook API migrated | WithCustomValidator to WithValidator (generics) |
| EVENT-RECORDER | Event recorder API changed | GetEventRecorderFor to GetEventRecorder |
| FAKE-CLIENT | Fake client deprecated | NewSimpleClientset: add SA1019 linter exclusion |
| STDLIB-MIGRATE | Utility moved to stdlib | slices.Contains, context package |
| CONTEXT-LEAK | Context cancel not deferred | lostcancel linter on context.WithTimeout |
| VALIDATION-MSG | Error message format changed | Kubernetes changed validation error strings |
| TIMEOUT-PERF | Slower fake client init | NewClientset builds type tables, causing timeouts |

Each pattern entry in the skill will include a grep command to detect it
and a code snippet showing the fix.

## Version Discovery

Three versions must be auto-discovered because they do not follow the
`k8s.io/api` version pattern:

**controller-runtime**: Historical mapping shows k8s 1.33 = CR v0.21,
1.34 = CR v0.22, 1.35 = CR v0.23. The pattern is
`v0.$((K8S_MINOR - 12)).x`. Verify against the Go module proxy before
using; fall back to `go list -m -versions` if the computed version is
not yet published.

**Go version**: Fetch from the upstream k8s go.mod at
`https://raw.githubusercontent.com/kubernetes/kubernetes/vX.Y.Z/go.mod`.

**structured-merge-diff major version**: Also from the upstream k8s
go.mod. The major version can change between k8s releases (v4 to v6
during the 1.34 rebase).

## Deliverables

Two files:

**`scripts/k8s-rebase.sh`** — standalone bash script, ~450 lines.
Handles all mechanical steps. Supports `--dry-run`. Can be run
independently without Claude Code. Features:

- Idempotency: detects if already at target version
- Resume: checks git log for existing commits, skips completed steps
- Version discovery: auto-detects controller-runtime, Go, and
  structured-merge-diff versions
- Commits: creates one commit per module plus codegen and version
  updates, matching the team's message format, all with `--signoff`

**`.claude/commands/k8s-rebase.md`** — Claude Code skill, ~600 lines.
Invokes the script for mechanical phases, then provides agent
instructions for build validation and fixups. Includes the breakage
pattern database as a reference section.

## Open Questions

1. **KIND version**: should it be bumped during a k8s rebase? Currently
   v0.27.0; new k8s versions may require a newer KIND release.
2. **controller-gen version**: currently v0.19.0, pinned in
   `update-codegen.sh` line 23. Should it be auto-discovered?
3. **Commit message convention**: the format varied between 1.34 (version
   in title) and 1.35 (no version in title). Which should the script use?
4. **Preemptive fixups**: the 1.35 rebase applied known fixes before the
   module updates. Should the script support a "pre-rebase fixups"
   phase, or leave that to the agent?
