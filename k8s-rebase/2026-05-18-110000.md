# OVN-Kubernetes K8s Rebase Automation

## Summary

When Kubernetes releases a new version, OVN-Kubernetes must bump all
`k8s.io/*` dependencies across its three Go modules. This is called
the "K8s rebase." Based on the 1.33-1.35 rebases, roughly 70% of the
work is mechanical: dependency updates, vendoring, code generation, and
version string replacements across CI and docs. The remaining 30% is
fixing build, lint, and test breakage caused by upstream API changes —
typically 100-150 lines of hand-written code.

The 1.35 rebase (PR #6099) touched 2,100 files with 163k lines of
churn (99k insertions, 64k deletions), but the human-authored fixup
commits totaled 114 lines. The rest was vendored dependencies,
generated code, and go.mod/go.sum changes.

This document describes automation that handles the mechanical portion
and guides an agent through the breakage fixes.

Historically, ovnk skipped k8s minor versions (1.27 → 1.30 → 1.33),
spacing rebases roughly a year apart. The 1.33 → 1.34 → 1.35 sequence
was tighter (2-5 months each). The frequency going forward depends on
whether the team continues tracking every minor release.

## Repository Structure

Three independent Go modules, no `go.work` file (as of May 2026):

| Module | Role | Vendors? | Validation |
| -------------------- | ---------------------- | -------- | -------------------- |
| `go-controller/` | Main ovnk controller | Yes | `make` / `make lint` / `make test` |
| `test/e2e/` | End-to-end tests | No | `go build ./...` |
| `test/conformance/` | NetworkPolicy conformance | No | `make conformance` |

The `go-controller/` vendor directory contains ~3,200 `k8s.io/*.go`
files. This is why the go-controller rebase commit alone changes
2,000+ files.

The module path is `github.com/ovn-kubernetes/ovn-kubernetes`
(changed from `ovn-org` as part of CNCF incubation, PR #5965).

## How a Rebase Actually Plays Out

The commit sequence from the 1.35 rebase (PR #6099) shows that fixups
are interleaved with module updates, not a separate phase:

```text
5238d48f7  Feb 26  disable WatchListClient feature gate for unit tests
4b266dedb  Feb 26  fix CNI utils test timeout after K8s 1.35 bump
d10c28ee8  Mar 11  test/e2e rebase
2f6c248f2  Mar 11  test/conformance rebase
32077814e  Mar 11  Updates kubernetes versions
88885c3a0  Mar 11  Fix e2e tests failure
4621b77f4  Mar 12  go-controller rebase
baa2c262d  Mar 12  update go mod tidy in e2e
9255baafc  Mar 13  Bump codegen
```

Key observations:

- **Preemptive fixups came first.** The WatchListClient and timeout
  fixes were committed before any module was rebased — the author
  knew these issues would arise from 1.35 and fixed them upfront.
- **Test modules were rebased before go-controller.** They are
  lighter-weight (no vendoring) and provide faster feedback.
- **The go-controller commit included lint fixes inline** — its
  commit message body documents 6 deprecation fixes alongside the
  `go get` recipe.

The automation maps these commits to the following phases.

## Automation Design

### Core Principle: Derive, Don't Hardcode

The go get recipes for the 1.34 and 1.35 rebases were 90% identical
but not 100% — conformance dropped 3 packages between rebases, and
`golang.org/x/tools` appeared in 1.34 but not 1.35. Hardcoding the
exact package list would require manual updates for every future
rebase.

Instead, the script derives what to bump by scanning each module's
`go.mod`:

1. **Version-pinned packages**: any `k8s.io/*` or `sigs.k8s.io/*`
   entry whose current version matches `v0.{OLD_K8S_MINOR}.*` gets
   bumped to `v0.{NEW_K8S_MINOR}.*`. This catches both direct and
   indirect deps (like `apiextensions-apiserver` and
   `controller-manager`, which are indirect but need explicit
   pinning).

2. **Pseudo-versioned packages**: entries at
   `v0.0.0-{timestamp}-{hash}` (like `k8s.io/kube-openapi`,
   `k8s.io/utils`, `github.com/openshift/api`) get `go get <pkg>`
   without a version pin to pull latest.

3. **Independent packages**: entries with their own versioning
   schemes (like `k8s.io/klog/v2` at v2.130.1 or
   `sigs.k8s.io/yaml` at v1.6.0) are skipped — `go mod tidy`
   resolves them.

4. **Special case — `k8s.io/kubernetes`**: uses the `v1.XX.Y`
   scheme instead of `v0.XX.Y`.

5. **Special case — `sigs.k8s.io/controller-runtime`**: uses its
   own versioning (v0.21 for k8s 1.33, v0.22 for 1.34, v0.23 for
   1.35 — offset of 12 from k8s minor). The script computes
   `v0.$((K8S_MINOR - 12)).0` and verifies it exists on the Go
   module proxy. If not, falls back to unversioned `go get`.

The same derivation logic applies to all three modules. After the
`go get` commands, each module runs `go mod tidy` (go-controller
also runs `go mod vendor`).

**Version reference files** are discovered the same way — by
grepping the repo for the old version string:

```bash
grep -rn "v${OLD_K8S_VERSION}" \
  --include="*.yml" --include="*.yaml" --include="*.sh" \
  --include="*.md" --include="Makefile*" . \
  | grep -v vendor | grep -v .git | grep -v go.mod
```

The 1.35 "Updates kubernetes versions" commit touched 10 files. A
grep for `v1.35.0` across the repo finds the same set. The script
does the grep, replaces in all hits, and lists them in the commit
message.

### Phase 1: Module Dependency Updates (3 commits)

For each module (test/e2e first, then test/conformance, then
go-controller):

1. Scan `go.mod` for packages to bump using the derivation rules
2. Run the computed `go get` commands
3. Run `go mod tidy` (and `go mod vendor` for go-controller)
4. Commit with `--signoff`, body listing every `go get` run

Commit message convention: the title names the module
(`"test/e2e rebase"`), the body lists every `go get` command.

For reference, the 1.35 recipes were 14 commands (test/e2e), 4
commands (conformance), and 16 commands (go-controller). The full
recipes from those commits are preserved in git history at
`d10c28ee8`, `2f6c248f2`, and `4621b77f4`.

### Phase 2: Code Generation (1 commit)

Two version pins in `go-controller/hack/update-codegen.sh`:

- Line 31: `k8s.io/code-generator/cmd/%s@v0.XX.Y` (must match k8s)
- Line 23: `controller-tools/cmd/controller-gen@v0.19.0` (may need
  independent bumping)

Update the code-generator version to match the new k8s, then run
`make -C go-controller codegen`. This regenerates ~90 files (CRD
clients, informers, listers, apply configurations). Commit as
`"Bump codegen"`.

### Phase 3: Version Reference Updates (1 commit)

Replace the old k8s version string with the new one in all files
discovered by grep (typically ~10 files across CI workflows,
scripts, and docs). If the Go version also changed (check upstream
k8s go.mod), update `GO_VERSION` in Makefiles and the
`go`/`toolchain` directives in all three go.mod files.

Commit as `"Updates kubernetes versions"` with the body listing
each file changed.

### Phase 4: Build Validation and Fixups (agent-guided)

After the mechanical steps, validate each module:

1. `cd go-controller && make` — build
2. `cd go-controller && make lint` — lint (catches deprecations)
3. `cd go-controller && make test` — unit tests
4. `cd test/e2e && go build ./...` — e2e build
5. `cd test/conformance && make conformance` — conformance build

For each failure, the agent should:

1. **Read the error message.** Most k8s rebase breakage produces
   clear compiler errors (`undefined: <OldFunction>`,
   `too many arguments`, `SA1019 deprecated`) that point directly
   to the fix.
2. **Check the k8s release notes.** The upstream changelog
   documents deprecated APIs, removed feature gates, and breaking
   changes. Search for the affected symbol.
3. **Consult the pattern hints below.** Common categories from
   past rebases can accelerate recognition.
4. **Fix and commit** each batch separately (e.g.,
   `"Fix e2e tests failure"`, `"Fix lint issues"`).

If a failure doesn't match any known pattern and the agent can't
determine the fix from the error and changelog, flag it for human
review.

## Breakage Pattern Hints

Common categories from the 1.33-1.35 rebases. These are starting
points for recognition, not an exhaustive catalog — each new k8s
release brings its own changes.

| Category | What breaks | How to fix |
| ----------------------------- | ------------------------------- | --------------------------------- |
| Feature gate default changed | Tests hang or panic | Disable the gate in test env |
| Function renamed | `undefined: <OldName>` | Search-replace + import update |
| Function signature changed | `too many/few arguments` | Add missing parameter (often a logger) |
| Admission API migrated | SA1019 on old webhook API | Switch to new generics-based API |
| Fake client deprecated | SA1019 on `NewSimpleClientset` | Add linter exclusion |
| Utility moved to stdlib | SA1019 on `k8s.io/utils/...` | Replace with stdlib equivalent |
| Validation message changed | Test assertion mismatch | Update expected error strings |

Concrete example — **WatchListClient** (1.35):

K8s 1.35 enabled `WatchListClient` by default. Fake clientsets from
third-party libraries do not support WatchList semantics, causing
informers to hang. The fix:

```bash
# In go-controller/hack/test-go.sh:
export KUBE_FEATURE_WatchListClient=false
```

```go
// In each *_suite_test.go importing k8s.io/kubernetes/pkg/features:
utilfeature.DefaultMutableFeatureGate.SetFromMap(map[string]bool{
    "WatchListClient": false,
})
```

## Deliverables

Two files:

**`scripts/k8s-rebase.sh`** — standalone bash script. Handles all
mechanical steps (Phases 1-3). Can be run independently without
Claude Code. Takes one argument: the target k8s version
(e.g., `1.36.0`). Features:

- Derives the go get package list from each module's go.mod
- Discovers version reference files via grep
- Auto-detects Go version from upstream k8s go.mod
- Auto-detects controller-runtime version (heuristic + proxy check)
- Creates one commit per phase, all with `--signoff`
- Exits early if already at target version (idempotent)

**`.claude/commands/k8s-rebase.md`** — Claude Code skill. Invokes
the script for Phases 1-3, then provides agent instructions for
Phase 4 (build validation and fixups). Includes the breakage pattern
hints and instructions for handling unknown failures.

## Open Questions

1. **controller-gen version**: currently v0.19.0 in
   `update-codegen.sh`. Should it be auto-discovered or manually
   specified as a script argument?
2. **Commit message convention**: 1.34 put the version in the title
   (`"go-controller rebase 1.34"`); 1.35 did not. Which should the
   script use?
3. **Preemptive fixups**: the 1.35 rebase applied known fixes before
   the module updates. Should the script support a pre-rebase fixup
   phase, or leave that to the agent?
