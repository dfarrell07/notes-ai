# OVN-Kubernetes K8s Rebase Automation

## Summary

When Kubernetes releases a new version, OVN-Kubernetes must bump all
`k8s.io/*` dependencies across its three Go modules. This is called
the "K8s rebase." Based on the 1.33-1.35 rebases, roughly 70% of the
work is mechanical: dependency updates, vendoring, code generation, and
version string replacements across CI and docs. The remaining 30% is
fixing build, lint, and test breakage caused by upstream API changes —
typically 100-150 lines of hand-written code.

The 1.35 rebase (PR #6099) touched 2,100 files with 163k lines of
churn (99k insertions, 64k deletions), but the human-authored fixup
commits totaled 114 lines. The rest was vendored dependencies,
generated code, and go.mod/go.sum changes.

This document describes automation that handles the mechanical portion
and guides an agent through the breakage fixes using a pattern database
built from past rebases.

Historically, ovnk skipped k8s minor versions (1.27 → 1.30 → 1.33),
spacing rebases roughly a year apart. The 1.33 → 1.34 → 1.35 sequence
was tighter (2-5 months each). The frequency going forward depends on
whether the team continues tracking every minor release.

## What the Repo Looks Like

Three independent Go modules, no `go.work` file (as of May 2026):

| Module | Role | Vendors? | Build command |
| -------------------- | ---------------------- | -------- | -------------------- |
| `go-controller/` | Main ovnk controller | Yes | `make` |
| `test/e2e/` | End-to-end tests | No | `go build ./...` |
| `test/conformance/` | NetworkPolicy conformance | No | `make conformance` |

The `go-controller/` vendor directory contains ~3,200 `k8s.io/*.go`
files. This is why the go-controller rebase commit alone changes
2,000+ files.

The module path recently changed from `github.com/ovn-org/ovn-kubernetes`
to `github.com/ovn-kubernetes/ovn-kubernetes` as part of CNCF
incubation (PR #5965).

## How a Rebase Actually Plays Out

The commit sequence from the 1.35 rebase (PR #6099) shows the real
workflow is not a clean pipeline — fixups are interleaved with module
updates:

```text
5238d48f7  Feb 26  disable WatchListClient feature gate for unit tests
4b266dedb  Feb 26  fix CNI utils test timeout after K8s 1.35 bump
d10c28ee8  Mar 11  test/e2e rebase
2f6c248f2  Mar 11  test/conformance rebase
32077814e  Mar 11  Updates kubernetes versions
88885c3a0  Mar 11  Fix e2e tests failure
4621b77f4  Mar 12  go-controller rebase
baa2c262d  Mar 12  update go mod tidy in e2e
9255baafc  Mar 13  Bump codegen
```

Key observations:

- **Preemptive fixups came first.** The WatchListClient and timeout
  fixes were committed before any module was rebased — the author
  knew these issues would arise from 1.35 and fixed them upfront.
- **Test modules were rebased before go-controller.** They are
  lighter-weight (no vendoring) and provide faster feedback.
- **The go-controller commit included lint fixes inline** — its
  commit message body documents 6 deprecation fixes alongside the
  `go get` recipe.
- **A cleanup pass followed** (`update go mod tidy in e2e`) to catch
  residual dependency drift after the go-controller vendor landed.

The automation maps these commits to the following phases.

## What to Automate

### Version Discovery (prerequisites)

Before any module is touched, three versions must be resolved because
they do not follow the `k8s.io/api` version pattern:

**controller-runtime**: k8s 1.33 used v0.21, 1.34 used v0.22, 1.35
uses v0.23. The pattern is `v0.$((K8S_MINOR - 12)).x`. Verify against
the Go module proxy before using; fall back to
`go list -m -versions` if the computed version is not yet published.

**Go version**: Fetch from the upstream k8s go.mod at
`raw.githubusercontent.com/kubernetes/kubernetes/vX.Y.Z/go.mod`. The
1.35 rebase required Go 1.24 → 1.25.

**structured-merge-diff major version**: Also from the upstream k8s
go.mod. The major version changed from v4 to v6 during the 1.34
rebase — the `go get` command referenced v4 but `go mod tidy` resolved
to v6.

### Mechanical Steps (deterministic, scriptable)

**Module dependency updates** — for each module, run a series of
`go get` commands to pin `k8s.io` packages to the new version, then
`go mod tidy` (and `go mod vendor` for go-controller).

The exact commands per module are taken from the 1.35 rebase commit
messages. Commit message convention: the title names the module
(`"test/e2e rebase"`), the body lists every `go get` command run.
The 1.34 rebase included the version in the title
(`"go-controller rebase 1.34"`); 1.35 did not.

**go-controller** (16 `go get` commands):

```text
go get k8s.io/api@v0.35.1
go get k8s.io/client-go@v0.35.1
go get k8s.io/component-helpers@v0.35.1
go get k8s.io/kubernetes@v1.35.1
go get k8s.io/apiextensions-apiserver@v0.35.1
go get sigs.k8s.io/controller-runtime@v0.23.1
go get k8s.io/controller-manager@v0.35.1
go get sigs.k8s.io/structured-merge-diff/v4@v4.7.0
go get sigs.k8s.io/knftables@v0.0.20
go get k8s.io/kubelet@v0.35.1
go get sigs.k8s.io/apiserver-network-proxy/konnectivity-client@v0.34.0
go get k8s.io/kube-openapi
go get sigs.k8s.io/json
go get k8s.io/utils
go get github.com/openshift/api
go get github.com/openshift/client-go
go mod tidy
go mod vendor
```

Note: `apiextensions-apiserver` and `controller-manager` are
`// indirect` in go.mod but must be explicitly pinned — `go mod tidy`
alone does not reliably resolve them to the correct version.
`openshift/api` and `openshift/client-go` are direct dependencies
used by the ovnk controller for OpenShift API types.

**test/e2e** (14 `go get` commands):

```text
go get k8s.io/kubernetes@v1.35.1
go get k8s.io/api@v0.35.1
go get k8s.io/client-go@v0.35.1
go get k8s.io/pod-security-admission@v0.35.1
go get k8s.io/kubectl@v0.35.1
go get sigs.k8s.io/controller-runtime@v0.23.1
go get k8s.io/mount-utils@v0.35.1
go get k8s.io/cri-client@v0.35.1
go get k8s.io/dynamic-resource-allocation@v0.35.1
go get k8s.io/kube-scheduler@v0.35.1
go get k8s.io/csi-translation-lib@v0.35.1
go get k8s.io/cloud-provider@v0.35.1
go get k8s.io/controller-manager@v0.35.1
go get k8s.io/kube-openapi
go mod tidy
```

Note: Many of these (mount-utils, cri-client, etc.) are transitive
dependencies that need explicit version pinning. Without explicit
`go get`, `go mod tidy` may resolve them to incompatible versions.

**test/conformance** (4 `go get` commands):

```text
go get k8s.io/kubernetes@v1.35.1
go get k8s.io/client-go@v0.35.1
go get k8s.io/kube-openapi
go get k8s.io/utils
go mod tidy
```

Note: This module uses `controller-runtime v0.22.1` while the other
two use `v0.23.1`. The rebase recipe did not explicitly bump it — `go
mod tidy` handles the resolution. The automation should not force
alignment.

**Code generation** — two version pins must be updated in
`go-controller/hack/update-codegen.sh`:

- Line 31: `k8s.io/code-generator/cmd/%s@v0.XX.Y` (must match k8s)
- Line 23: `sigs.k8s.io/controller-tools/cmd/controller-gen@v0.19.0`
  (may need bumping independently)

Then run `make -C go-controller codegen`. This regenerates ~90 files
(CRD clients, informers, listers, apply configurations).

**Version reference updates** — sed-based replacements using the old
version as the search pattern (not line numbers, which shift between
releases):

K8S_VERSION (7 files):

- `.github/workflows/test.yml`
- `.github/workflows/performance-test.yml`
- `contrib/kind-common.sh`
- `test/scripts/install-kind.sh`
- `test/scripts/upgrade-ovn.sh`
- `docs/developer-guide/local_testing_guide.md`
- `docs/installation/INSTALL.KUBEADM.md`

Go version (only if changed, 6 files):

- `go-controller/Makefile` (GO_VERSION)
- `dist/images/Makefile` (GO_VERSION)
- `.github/workflows/docker.yml` (BUILDER_IMAGE)
- 3 `go.mod` files (`go` + `toolchain` directives)

Docs (2 files):

- `docs/features/requirements.md` (version matrix)
- `docs/installation/launching-ovn-kubernetes-on-kind.md`

### Agent-Guided Steps (require judgment)

After the mechanical steps, the agent validates each module:

1. Build go-controller: `cd go-controller && make`
2. Lint go-controller: `cd go-controller && make lint`
3. Run go-controller unit tests: `cd go-controller && make test`
4. Build test/e2e: `cd test/e2e && go build ./...`
5. Build test/conformance: `cd test/conformance && make conformance`

Failures are matched against the breakage pattern database below.
Unmatched failures are flagged for human review. Each batch of fixes
gets its own commit (e.g., `"Fix e2e tests failure"`,
`"Fix lint issues"`).

## Breakage Pattern Database

10 patterns from analyzing the 1.33-1.35 rebases. Each pattern in the
skill will include a grep command to detect it and a code snippet
showing the fix.

| ID | Pattern | 1.35 Example |
| --------------- | ------------------------------ | -------------------------------------------- |
| FG-DEFAULT | Feature gate default changed | WatchListClient hangs fake informers |
| FUNC-RENAME | Function renamed | WaitForServiceEndpointsNum → WaitForEndpointCount |
| SIG-CHANGE | Function signature changed | IsNodeReady now requires a logger parameter |
| ADMISSION-API | Admission webhook API migrated | WithCustomValidator → WithValidator (generics) |
| EVENT-RECORDER | Event recorder API changed | GetEventRecorderFor → GetEventRecorder |
| FAKE-CLIENT | Fake client deprecated | NewSimpleClientset: add SA1019 linter exclusion |
| STDLIB-MIGRATE | Utility moved to stdlib | slices.Contains, context package |
| CONTEXT-LEAK | Context cancel not deferred | lostcancel linter on context.WithTimeout |
| VALIDATION-MSG | Error message format changed | Kubernetes changed validation error strings |
| TIMEOUT-PERF | Slower fake client init | NewClientset builds type tables, causing timeouts |

Concrete example — **FG-DEFAULT** (WatchListClient in 1.35):

K8s 1.35 enabled `WatchListClient` by default. Fake clientsets from
third-party libraries do not support WatchList semantics, causing
informers to hang waiting for bookmark events. The fix has two parts:

```bash
# In go-controller/hack/test-go.sh, add:
export KUBE_FEATURE_WatchListClient=false
```

```go
// In each *_suite_test.go that imports k8s.io/kubernetes/pkg/features:
utilfeature.DefaultMutableFeatureGate.SetFromMap(map[string]bool{
    "WatchListClient": false,
})
```

Affected 5 files in go-controller (test-go.sh + 4 suite_test.go files).

## Deliverables

Two files:

**`scripts/k8s-rebase.sh`** — standalone bash script (~450 lines).
Handles all mechanical steps. Can be run independently without Claude
Code. Features:

- `--dry-run`: prints all commands without executing
- Idempotency: detects if already at target version
- Resume: checks git log for existing commits, skips completed steps
- Version discovery: auto-detects controller-runtime, Go, and
  structured-merge-diff versions
- Commits: one per module plus codegen and version updates, matching
  the team's message format, all with `--signoff`

**`.claude/commands/k8s-rebase.md`** — Claude Code skill (~600 lines).
Invokes the script, then provides agent instructions for build
validation and fixups. Includes the breakage pattern database as a
reference section.

## Open Questions

1. **KIND version**: should it be bumped during a k8s rebase? Currently
   v0.27.0; new k8s versions may require a newer KIND release.
2. **controller-gen version**: currently v0.19.0 in
   `update-codegen.sh`. Should it be auto-discovered or manually
   specified?
3. **Commit message convention**: 1.34 included the version in the
   title (`"go-controller rebase 1.34"`); 1.35 did not. Which should
   the script use?
4. **Preemptive fixups**: the 1.35 rebase applied known fixes before
   the module updates. Should the script support a pre-rebase fixup
   phase, or leave that to the agent?
