# OVN-Kubernetes K8s Rebase Automation

## Summary

When Kubernetes releases a new version, OVN-Kubernetes must bump all
`k8s.io/*` dependencies across its three Go modules. This is called
the "K8s rebase." Based on the 1.33-1.35 rebases, the bulk of the
work is mechanical: dependency updates, vendoring, code generation,
and version string replacements across CI and docs. The remaining
effort — fixing build, lint, and test breakage caused by upstream
API changes — is small by volume but requires engineering judgment.

The 1.35 rebase (PR #6099) touched 2,100 files with 163k lines of
churn, but the human-authored code changes totaled ~160 lines across
fixup commits and inline lint fixes. The rest was vendored
dependencies, generated code, and go.mod/go.sum changes.

This document describes automation that handles the mechanical portion
and guides an agent through the breakage fixes. Target: the upstream
repo (`ovn-org/ovn-kubernetes`). Downstream
(`openshift/ovn-kubernetes`) receives the rebase via its separate
merge process.

The mechanical phases take minutes (network + codegen). The fixup
phase depends on breakage, but based on the 1.35 rebase (~160
lines of fixes), an agent-guided session should take hours. The
Prior Art durations (1-3 weeks) reflect calendar time including
review and CI — the actual engineering work in the 1.35 rebase
was concentrated in 4 days. The automation targets that focused
time, not the review cycle.

## Repository Structure

Three independent Go modules, no `go.work` file (as of May 2026):

| Module | Role | Vendors? | Validation |
| -------------------- | ---------------------- | -------- | -------------------- |
| `go-controller/` | Main ovnk controller | Yes | `make` / `make lint` / `make test` |
| `test/e2e/` | End-to-end tests | No | `go build ./...` |
| `test/conformance/` | NetworkPolicy conformance | No | `make conformance` |

The `go-controller/` vendor directory contains ~3,200 `k8s.io/*.go`
files. This is why the go-controller rebase commit alone changes
2,000+ files.

The module path is `github.com/ovn-kubernetes/ovn-kubernetes`
(changed from `ovn-org` as part of CNCF incubation, PR #5965).

## How a Rebase Actually Plays Out

The commit sequence from the 1.35 rebase (PR #6099) shows that fixups
are interleaved with module updates, not a separate phase:

```text
5238d48f7  Feb 26  disable WatchListClient feature gate for unit tests
4b266dedb  Feb 26  fix CNI utils test timeout after K8s 1.35 bump
d10c28ee8  Mar 11  test/e2e rebase
2f6c248f2  Mar 11  test/conformance rebase
32077814e  Mar 11  Updates kubernetes versions
88885c3a0  Mar 11  Fix e2e tests failure
4621b77f4  Mar 12  go-controller rebase
baa2c262d  Mar 12  update go mod tidy in e2e
9255baafc  Mar 13  Bump codegen
```

Key observations:

- **Preemptive fixups came first.** The WatchListClient and timeout
  fixes were committed before any module was rebased — the author
  knew these issues would arise from 1.35 and fixed them upfront.
- **Test modules were rebased before go-controller.** They are
  lighter-weight (no vendoring) and provide faster feedback.
- **The go-controller commit included lint fixes inline** — its
  commit message body documents 6 deprecation fixes alongside the
  `go get` recipe.

The automation maps these commits to the following phases.

## Automation Design

### Core Principle: Best-Effort Derivation + Build Safety Net

The go get recipes changed between the 1.34 and 1.35 rebases —
conformance dropped 3 packages, `golang.org/x/tools` appeared
then disappeared. Hardcoding the exact package list would require
manual updates for every future rebase.

Instead, the script derives what to bump by scanning each module's
`go.mod`. The derivation is deliberately best-effort: it catches
the majority of packages, and Phase 4 (build validation) catches
whatever it misses. This is robust because a missed `go get`
produces a clear build error that the agent can diagnose and fix.

**Derivation rules** — these are generic to any Go project that
consumes `k8s.io` dependencies, not ovnk-specific. The script
structure and phase ordering are ovnk-specific; the derivation
algorithm is portable. Validated by simulation against the 1.34
and 1.35 rebases: ~97% of go get commands correctly predicted,
zero false positives. Remaining ~3% are caught by `go mod tidy`
or Phase 4.

Rules (applied only to `k8s.io/*`, `sigs.k8s.io/*`, and
`github.com/openshift/*` entries):

1. **Version-locked**: current version matches
   `v{N}.{OLD_MINOR}.*` → bump to `v{N}.{NEW_MINOR}.*`. Handles
   both `v0.35.*` (most k8s.io packages) and `v1.35.*`
   (k8s.io/kubernetes) with the same rule. Catches direct and
   indirect deps alike.

2. **controller-runtime**: version is independent (v0.23.1 for
   k8s 1.35). Compute `v0.$((K8S_MINOR - 12)).0`, verify on the
   Go module proxy, fall back to rule 3. Formula confirmed correct
   across 3 consecutive rebases.

3. **Everything else in k8s ecosystem**: `go get <pkg>` without a
   version pin (latest). Catches pseudo-versioned packages
   (kube-openapi, utils, openshift/api), independently-versioned
   packages (klog, yaml, knftables, network-policy-api,
   structured-merge-diff), and any new ecosystem deps. Extra go
   gets are fast no-ops for already-current packages. If latest is
   incompatible, the build catches it in Phase 4.

Simulation shows this 3-rule approach catches ~97% of go get
commands from the 1.34 and 1.35 rebases (up from ~85% with the
previous 4-rule design that skipped independently-versioned
packages).

**Version reference files** are discovered by grep. Two passes
are needed because some files use the full version (`v1.35.0`)
while others use the short form (`1.35` or `v1.35`):

```bash
# Full version (CI workflows, scripts)
grep -rn "v${OLD_K8S_FULL}" \
  --include="*.yml" --include="*.yaml" --include="*.sh" \
  --include="*.md" --include="Makefile*" . \
  | grep -v vendor | grep -v .git | grep -v go.mod

# Short version (docs — requirements.md, INSTALL.KUBEADM.md)
grep -rn "\b${OLD_K8S_MAJOR_MINOR}\b" --include="*.md" docs/
```

The 1.35 "Updates kubernetes versions" commit touched 10 files
using this approach. Go version changes are handled separately
by comparing go.mod `go` directives before and after Phase 1.

### Phase 0: Prerequisites

Check all prerequisites upfront before modifying anything. On
failure, print what's wrong and how to fix it, then exit 1.

1. Required tools: `go`, `git`, `make`, `curl`, `sed`, `grep`
2. Go version meets the upstream k8s requirement (fetch the
   required version from `kubernetes/kubernetes@vX.Y.Z/go.mod`)
3. Target k8s version is published on the Go module proxy
4. Clean git working tree (`git status --porcelain` is empty)
5. Create branch: `git checkout -b bump${K8S_MAJOR_MINOR}`
   (convention from PR #6099: `bump1.35`, not `bump1.35.1`)

### Phase 1: Module Dependency Updates (4 commits)

For each module (test/e2e first, then test/conformance, then
go-controller):

1. Scan `go.mod` for packages to bump using the derivation rules
2. Run the computed `go get` commands
3. Run `go mod tidy` (and `go mod vendor` for go-controller)
4. For go-controller: run `hack/verify-go-mod-vendor.sh`
5. Commit with `--signoff`; title includes module and k8s version
   (e.g., `"test/e2e rebase 1.36"`), body lists every `go get`
   command. The repo has no enforced commit message format
   (CONTRIBUTING.md requires DCO sign-off only).

Note: `go get` and `go mod tidy` automatically bump the `go` and
`toolchain` directives in go.mod when the new dependencies require
a newer Go version. No manual editing is needed.

**After go-controller is done, re-run `go mod tidy` in test/e2e.**
The test/e2e module depends on go-controller via a replace
directive (`../../go-controller`), so its indirect dependency
versions are resolved through go-controller's module graph. When
go-controller's rebase changes those indirects, test/e2e needs a
second tidy to re-sync. This is why the 1.35 rebase had a
cleanup commit (`baa2c262d: update go mod tidy in e2e`).
Conformance has no go-controller dependency, so it does not need
this step. Commit the test/e2e re-tidy separately.

For reference, the 1.35 recipes are in git at `d10c28ee8`,
`2f6c248f2`, `4621b77f4`, and `baa2c262d`.

### Phase 2: Code Generation (1 commit)

Two version pins in `go-controller/hack/update-codegen.sh`
(find via `grep "code-generator\|controller-gen"` — do not rely
on line numbers as they shift between releases):

- `k8s.io/code-generator/cmd/%s@v0.XX.Y` (must match k8s)
- `controller-tools/cmd/controller-gen@v0.19.0` (may need
  independent bumping)

Update the code-generator version to match the new k8s, then run
`make -C go-controller codegen`. This regenerates ~90 files (CRD
clients, informers, listers, apply configurations). Commit as
`"Bump codegen"`.

### Phase 3: Version Reference Updates (1 commit)

Replace the old k8s version string with the new one in all files
discovered by grep (typically ~10 files across CI workflows,
scripts, and docs). If the Go version also changed (determined by
comparing the old and new go.mod `go` directives — already updated
automatically in Phase 1), update `GO_VERSION` in Makefiles and
Go image tags in CI workflows (e.g., `golang:1.25` in
`docker.yml`). The go.mod files themselves are already correct
from Phase 1.

Commit as `"Updates kubernetes versions"` with the body listing
each file changed.

### Phase 4: Build Validation and Fixups (agent-guided)

Phase 4 follows the **"Small World, Many Agents"** pattern: use
scripts to collect and categorize errors deterministically, then
give the agent focused context for each problem. The agent
evaluates pre-collected evidence — it never searches.

**Step 1: Collect (deterministic).** The script runs all
validation commands, capturing output to files:

```bash
make -C go-controller 2>&1 | tee /tmp/rebase-build.log
make -C go-controller lint 2>&1 | tee /tmp/rebase-lint.log
make -C go-controller test 2>&1 | tee /tmp/rebase-test.log
(cd test/e2e && go build ./...) 2>&1 | tee /tmp/rebase-e2e.log
(cd test/conformance && make conformance) 2>&1 | tee /tmp/rebase-conformance.log
```

**Step 2: Categorize (deterministic).** Parse the logs to extract
actionable errors — file paths, error types, affected symbols.
Group by category (build errors, lint SA1019, test failures).
Write a structured summary the agent can act on without wading
through raw output.

**Step 3: Fix (agent-guided).** For each error category, the
agent gets:

- The extracted errors (not the full log — just the actionable
  lines)
- The affected source files
- The relevant subset of `docs/k8s-rebase-patterns.md`

The agent reads the error, checks the k8s release notes for the
affected symbol, applies the fix, and commits. Each category gets
its own **independently revertable** commit (e.g.,
`"Fix e2e tests failure"`, `"Fix lint issues"`). If a fix
introduces a regression, reverting that single commit should not
undo other fixes.

**Step 4: Re-validate (deterministic).** After each fix batch,
the SCRIPT (not the agent) re-runs the failing validation step.
If it passes, move to step 5. If it still fails, the agent gets
the new errors and returns to step 3.

**Step 5: Antagonistic review (agent-guided).** After a fix
commit passes validation, a SEPARATE agent reviews the diff with
fresh context (no memory of how the fix was derived). This review
agent gets only:

- The git diff of the fix commit
- The original error it was meant to address
- The k8s release notes for the affected symbols

It answers one question: "Does this fix correctly address the
error without introducing side effects?" If the review agent
flags a concern, the fix is reverted and the original fix agent
retries with the review feedback added to its context. This
catches semantic errors that the build can't — e.g., a function
rename that compiles but maps arguments incorrectly, or a feature
gate disable that masks a real test failure.

**Iterate** steps 3-5 until all validation passes, with two
safeguards for unattended execution:

- **Bounded retries**: if the same error has been fixed, reverted,
  and retried 2 times, escalate to human review instead of
  looping indefinitely.
- **Clean context on retry**: the fix agent gets only the latest
  review feedback plus the original error, not the full history
  of failed attempts. Accumulated failure context confuses rather
  than helps.

If a failure doesn't match any known pattern and neither agent can
determine the fix, flag it for human review. The automation is
designed to run unattended — human intervention is the exception,
not the default.

If a fix requires a go.mod change (e.g., adding a missing `go get`
that the derivation missed), run `go mod tidy` (and vendor if
applicable) after the change.

## Breakage Pattern Hints

These will live in `docs/k8s-rebase-patterns.md` in the upstream
repo so the team can update them after each rebase. Common
categories from the 1.33-1.35 rebases — starting points for
recognition, not an exhaustive catalog:

| Category | What breaks | How to fix |
| ----------------------------- | ------------------------------- | --------------------------------- |
| Feature gate default changed | Tests hang or panic | Disable the gate in test env |
| Function renamed | `undefined: <OldName>` | Search-replace + import update |
| Function signature changed | `too many/few arguments` | Add missing parameter (often a logger) |
| Admission API migrated | SA1019 on old webhook API | Switch to new generics-based API |
| Fake client deprecated | SA1019 on `NewSimpleClientset` | Add linter exclusion |
| Utility moved to stdlib | SA1019 on `k8s.io/utils/...` | Replace with stdlib equivalent |
| Validation message changed | Test assertion mismatch | Update expected error strings |

Concrete example — **WatchListClient** (1.35): k8s 1.35 enabled
this feature gate by default, but fake clientsets don't support
WatchList semantics, causing informers to hang. Fix: disable the
gate via env var in `test-go.sh` and via `SetFromMap` in 4
`*_suite_test.go` files (5 files total, see commit `5238d48f7`).

## Deliverables

Following the proven pattern (thin skill → orchestrator scripts
→ prompt templates → reference docs):

### In the upstream repo (`ovn-org/ovn-kubernetes`)

**`go-controller/hack/k8s-rebase.sh`** — main orchestrator
script. Handles Phases 0-3 (deterministic). Can run standalone
without Claude Code. Takes one argument: the target k8s version
(e.g., `1.36.0`). Features:

- Checks prerequisites upfront with clear error messages
- Derives go get packages from go.mod (3 derivation rules)
- Discovers version reference files via grep
- Auto-detects Go version and controller-runtime version
- Creates 6 commits, all with `--signoff`
- Idempotent (exits early if already at target version)
- Safe to re-run after partial failure: `go get` and `go mod tidy`
  are no-ops for already-bumped packages, commits are only created
  when there are staged changes
- Exit codes: 0 = done, 1 = error, 2 = Phase 4 needed
- On exit 2, prints structured summary (context for the agent)
- Rollback: dedicated branch, main never modified

**`go-controller/hack/k8s-rebase-validate.sh`** — Phase 4
steps 1-2 (deterministic). Runs build, lint, and test for all
modules. Captures output to `/tmp/rebase-*.log`. Parses logs to
extract actionable errors. Writes a categorized summary (build
errors, lint SA1019, test failures) with file paths and symbols.
This crafted output IS the agent's context — the agent evaluates
pre-collected evidence, never searches.

**`go-controller/hack/k8s-rebase-review-prompt.md`** — prompt
template for Phase 4 step 5 (antagonistic review). Contains
`${VARIABLE}` placeholders filled by `review.sh` via `envsubst`.
The review agent gets ONLY:

- `${DIFF}` — the git diff of the fix commit
- `${ORIGINAL_ERROR}` — the error it was meant to address
- `${K8S_CHANGELOG}` — relevant k8s release notes excerpt
- `${PATTERN_HINT}` — matching entry from patterns file, if any

It answers one question: "Does this fix correctly address the
error without side effects?" Output: `APPROVE: <reason>` or
`REJECT: <reason>`. No memory of how the fix was derived — fresh
context only.

**`go-controller/hack/k8s-rebase-review.sh`** — Phase 4 step 5
orchestrator. For each fix commit: loads the prompt template,
substitutes variables with pre-fetched evidence, invokes
`claude -p` (the review agent as a separate process with no
shared context), parses the verdict. On REJECT: reverts the
commit and feeds the rejection reason back to the fix agent
(latest feedback only, not failure history).

**`docs/k8s-rebase-patterns.md`** — breakage pattern reference.
Contains the pattern hints table and concrete examples. Read by
the skill at Phase 4 start and by `review.sh` for pattern
matching. Updatable by anyone on the team after each rebase.

### In the workspace (`~/ovnk/`)

**`.claude/skills/k8s-rebase/SKILL.md`** — the invocable skill
entry point. A skill (not a command): skills are active
executables with metadata; commands are reference docs. SKILL.md
has frontmatter:

```yaml
---
name: k8s-rebase
description: Rebase OVN-Kubernetes to a new k8s version
argument-hint: "<version> (e.g., 1.36.0)"
user-invocable: true
allowed-tools: Bash, Read
---
```

Phase 2 skill (mixed md/sh — the maturity level where most
skills start, per the Pseudocode as Prototype pattern). Contains:

- **Bash blocks**: call `k8s-rebase.sh` (Phases 0-3), call
  `k8s-rebase-validate.sh` (collect/categorize errors), call
  `k8s-rebase-review.sh` (antagonistic review per fix commit)
- **Prose instructions**: guide the fix agent through each error
  category — read the extracted errors, consult the patterns
  file, apply the fix, commit with the right message format

Estimated ~60-80 lines (well under the ~1,000-line threshold
where proven skills become unmaintainable and need script
extraction). Designed to run unattended — human review is the
fallback for unrecognized failures.

## Open Questions

1. **controller-gen version**: currently v0.19.0 in
   `update-codegen.sh`. Changed in 2 of 3 recent rebases (v0.16.4
   → v0.19.0), with no predictable pattern tied to k8s versions.
   The script cannot derive it mechanically. Proposed default:
   accept an optional `--controller-gen-version` argument; if
   omitted, keep the current version and let the build surface any
   incompatibility.
