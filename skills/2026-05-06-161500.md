# Contributing a generic cve-fix to openshift/ai-helpers

Derive a generic Go CVE remediation skill from the Submariner-specific
implementation in submariner-io/shipyard. Contribute it to ai-helpers so
any OpenShift team can use it.

## The shipyard implementation

Shipyard has a working Claude plugin (v1.0.0) with an end-to-end CVE fix
pipeline in ~1,000 lines of bash (11 scripts including test helpers, plus
a review prompt for the Claude subagent):

1. **Detect** — validate repo/git, find go.mod files, detect container
   runtime and Grype, create fix branch
2. **Scan** — run Grype, collect all CVEs as JSON
3. **Fix** — `go get package@highest-fixed-version` across all go.mod
   files; separate path for stdlib (updates Go directive); block fixes
   that would bump Go or K8s minor version on stable branches
4. **Review** — Claude subagent handles edge cases: alternative versions,
   .grype.yaml ignore with justification
5. **Output** — ready-to-run `gh pr create` command

## Compared to compliance:analyze-cve

| | analyze-cve | cve-fix |
|---|---|---|
| Question | "Is this CVE exploitable?" | "Fix all CVEs in this repo" |
| Input | Single CVE ID | Repo + branch |
| Discovery | User provides CVE | Grype scans for all |
| Depth | Call graph reachability | Pragmatic — fix exists, apply it |
| Implementation | Pure markdown prompts | Bash scripts |

Complementary: analyze-cve investigates, cve-fix remediates. Neither
touches Jira.

## What needs generalizing

~80% of the code is already project-agnostic. The Submariner-specific parts:

| What | Where | Generalize how |
|------|-------|----------------|
| `0.23` → `release-0.23` branch naming | detect.sh | Drop or parameterize |
| Shipyard build image Go version check | detect.sh, fix-stdlib.sh | Skip when not present |
| `make BUILD_UPX=false build` before scan | scan.sh | Already conditional, gate better |
| `make unit` / `make clean` | fix-all.sh | Default to `go test ./...`, detect Makefile |
| K8s client-go minor version guard | fix-package.sh | Keep as opt-in (useful for any K8s project) |
| Toolchain directive stripping | lib.sh | Make configurable |
| Dapper container cleanup | fix-all.sh | Remove or use generic pattern |
| PR command fork/branch assumptions | fix-all.sh | Simplify |
| Submariner examples in docs | SKILL.md, review-prompt.md | Replace with generic examples |

Approach: project config via `.cve-fix.conf` sourced at the top of
detect.sh. Core scripts stay generic, projects override defaults.

## ai-helpers contribution requirements

Plugin structure (see `plugins/hello-world/` as canonical minimal example):

```
plugins/{name}/
├── .claude-plugin/plugin.json   # name, version, author, description
├── commands/{cmd}.md            # user-facing slash commands
├── skills/{name}/SKILL.md       # supporting sub-skills
├── skills/{name}/scripts/       # helper scripts
├── reference/                   # context docs
└── README.md
```

Key rules:
- Author in plugin.json should be `"github.com/openshift-eng"`
- Command .md: YAML frontmatter (`description:`, `argument-hint:`) plus
  man-page-style sections (Name, Synopsis, Description, Implementation,
  Return Value, Examples, Arguments at minimum) — `make lint` validates
- Bump version in plugin.json for any code change (CI enforces)
- `make lint` and `make update` before submitting
- Don't reference real people by name in skill content
- Check `/utils:review-ai-helpers-overlap` first
- Use `lower-case-with-dashes` for names; new plugin only when no existing one fits

Script-heavy skills are established: jira has ~850 lines Python+bash,
must-gather has ~2,790 lines Python across 9 analyzer scripts.
